# Project Rules - Staging

1 回観測のみの候補ルール。次回 incremental 抽出走行（incremental extraction run — `--from-conversation` / `--from-pr` / `--update`）で再観測されたら canonical へ promote されます。手動で `.local.md` へ移動することも可能（promote 待たずに採用する場合）。

## Project-specific patterns

- **Implicit observation counter via staging-entry presence (vs explicit counter file)**: 観測回数 N 回 gating 機構を導入時、明示 counter file（`observation-counts.json` 等の sidecar metadata）でなく **staging 領域に entry が存在する = 1 回観測済** という暗黙方式を優先する。判断軸: (i) staging directory に rule 以外の state file が増えると backup / merge-rules / apply-rules 連携で扱いが複雑化、(ii) 暗黙方式は staging file 自身が canonical state で記録系の二重管理が発生しない、(iii) N ≥ 3 等の閾値調整は YAGNI（現状 threshold = 2 で十分）。明示 counter 案は「将来 N ≥ 3 で閾値調整したい」motivation が surface した時点で再検討（現時点では over-engineering）
- **Move atomicity ordering for promote operations: canonical-append-first → verify → staging-delete-second**: staging → canonical へ promote する mechanism では順序を `(a) canonical append → (b) verify canonical write succeeded → (c) staging delete` で固定。失敗時の duplicate-vs-loss trade-off は **duplicate 側を選ぶ**（(a) 後の失敗は canonical entry intact / staging entry 残存 → 次走行の canonical-match skip で自然解消）。逆順（先に staging delete → canonical write 失敗）は loss 経路で復旧不能。`Edit` ベース staging delete は `old_string` not-unique で fail しても「duplicate を残して次走行解消」を docstring に明記
- **Per-mode read/write/promote matrix for incremental-staging mechanisms**: skill が複数 mode（Full / Update / Conversation / PR Review / Restructure / Compaction 等）を持ち、新規 gating 機構を一部 mode のみに適用する設計では、**mode × {Reads staging? / Writes staging? / Promotes on match?}** の 3 axis matrix を Mode interaction summary で表形式 documenting。表があると後続 mode 追加時の整合性 check が表 1 行追加で済む。canonical 配置: SKILL.md の `## Mode interaction summary` 単一 section（references/*.md からは安定 heading で cross-ref）。`Edge case — <mode> over a pre-populated staging directory` のような mode 特有 corner case は表外段落で補足
- **Two-axis dedup criterion: signature byte-equal/semantic AND context-phrase aligned, with single-axis edge cases enumerated**: staging → canonical の promote 判定のような dedup 機構では **2-axis 判定**（(a) inline code signature byte-equal **or** semantic-equivalent、(b) context phrase semantically aligned）を canonical、**single-axis match の edge cases を closed list で enumerate**: (a)-only → same-observation promote with overwritten context、(b)-only → not a match, fall through to new-staging-append。両 axis match 時の **どちらの context phrase が canonical に landed するか**（current observation = authoritative refresh / staging-side = older draft）も明記。axis を 1 つにすると false positive（signature 同じだが意味が異なる）/ false negative（同じ pattern を別表現で書いた 2 回目を miss）の両方が surface
- **Scope discipline: limit new gating mechanism to highest-signal category for initial subtask + explicit escalation path to subsequent subtask**: 抽出 / gating / consolidation 等の新 mechanism を導入時、初動 subtask では **bloat signal が最も強い 1 category のみに scope** し、language / framework / integration への拡張は subsequent subtask に defer。判断軸: (i) 実測 bloat signal（例: `project.rules.local.md` 41k 文字突出）で primary target が明確、(ii) 全 category 同時導入は tree complexity が線形以上に増加、(iii) escalation path（mirrored tree / single staging file + category prefix bullet 等 alternative）を Decisions に future option として記載すれば subtask N+1 で choice 取り直し容易。逆に初動から全 category gating は dead-on-arrival mechanism を量産する risk
- **Multi-heuristic-set subagent dispatch: distinct `--- LABEL ---` fences routed to distinct output arrays** (class-level extension of `Subagent dispatch prompt body lives in references/<mode>-prompt.md` fence-label convention): 1 dispatch が **複数 heuristic set** を持ち各 set の出力先 array が異なる場合（例: `--compact` の compaction heuristics → `mechanical_edits` / `structural_notes`、consolidation heuristics → `consolidation_proposals`）、各 heuristic set を **独立 `--- LABEL ---` fence** に分けて payload 構成。canonical: (i) fence label を heuristic set ごとに distinct（`--- COMPACTION HEURISTICS ---` / `--- CONSOLIDATION HEURISTICS ---`）、(ii) `--- COMPACTOR PROMPT ---` 等の subagent instruction 内で「two-heuristic-set / distinct-output-array routing」を明示、(iii) gating constants（`--- TARGET CHARS ---` / `--- MIN CLUSTER SIZE ---`）も独立 fence で渡し set 間の混線防止、(iv) `references/<mode>-prompt.md` § Contract に各 heuristic set → output array の mapping を canonical 配置。1 fence に複数 set を詰め込むと subagent が routing を誤判定し output array 取り違え（class-level extension audit 対象: 既存 1-fence / 1-set 設計を multi-set 化する際は全 dispatch site sweep）
- **Iter-1-only capture discipline for caller-judgment output arrays** (class-level extension of `verify-diff`'s `inferred_intent` persistence rule): Pattern A iteration loop で **caller-judgment 用の sibling output array**（`structural_notes` / `consolidation_proposals` 等、Edit で適用されない判断材料 array）が複数共存する場合、各 array は **iter 1 の値のみを captured value として保持**、iter 2+ の同 array は ignored（mutated content に対する re-emit は drift する）。canonical: (i) 全 sibling array に「captured from iter 1 only」明記、(ii) iter 1 で parseable verdict 無しなら `[]` で初期化、(iii) 上位 status mapping は captured values を読む、(iv) subagent prompt に「Do not re-emit consolidation_proposals on iter 2」instruction、(v) main thread も iter 2+ の同 array 値を consume しない。`inferred_intent` 単一 field の persistence を array 系 sibling output に拡張する class-level rule（applied 系 `mechanical_edits` / `suggested_edits` は iter 累積で別扱い、iter-1-only は **non-applied caller-judgment** 用）
- **N-way OR status mapping extension surfaces latent silent-drop bugs in sibling arrays**: top-level `status` mapping を単一 arm（例: `applied_edits_count > 0`）から **N-way OR over multiple sibling output arrays** に拡張する設計判断時、既存 arm の単独条件が **silent-drop 経路を内包していないか必ず audit**。canonical 例: `compacted` を `applied_edits_count > 0` から `(applied_edits_count > 0) OR (non-empty consolidation_proposals[]) OR (non-empty structural_notes[])` に拡張すると、`structural_notes` 単独 case が `no-actionable` に倒れて caller が `compacted` 分岐で取り溢す latent bug が incidentally 修正される。CHANGELOG / commit message に「incidentally fixes a latent bug where `<field>`-only files previously fell into `<wrong-status>` and were silently dropped by callers branching on `<right-status>`」と signal。逆 `no-actionable` 条件は **closed-list ALL-quantifier**（every file satisfies all of (...)）形に明示記述、boolean grouping 曖昧性を排除（vacuous-truth gap rule との交差）
- **Forward-compat schema validation for new optional output fields: missing key → empty, explicit non-array → schema-violation**: 既存 subagent dispatch の per-iter response schema に **新 optional output array** を追加する際、forward-compat の canonical 2-leg 判定: (i) **missing** the new key（older prompt が emit しない）→ main thread 側で **empty array として扱う**（schema violation にしない）、(ii) **explicitly present but non-array value**（malformed entry）→ **schema-violation で reject**。理由: older subagent prompt との roll-forward 互換性確保（callee SKILL.md 改修ラグ吸収）+ malformed entry の silent-pass 防止。SKILL.md 該当 parse step に「For forward-compat with older subagent prompts that do not emit `<field>`, the main thread treats a **missing** `<field>` key (not present in the JSON) as an empty array; only an explicitly non-array value triggers the schema-violation path」と 1 文 inline 明記。新 optional field 追加時の class-level extension audit 対象（全 dispatch site の parse step を同 form で sweep）
- **Routine non-interactivity preference: default to auto-apply mechanization over user-judgment gates for repeating routines**: 非対話 / ルーチン用途（`--compact` mode / `dev-workflow` `hooks.on_complete` / `dev-workflow-triage` 等）で subagent 出力が `caller-judgment notes` / `detection-only` / `structural notes` として user-gate 経由になっている場合、user feedback「ユーザー判断無しで作業を完了できるようにしたい」が出る前提として **auto-apply mechanization が default direction**。新規 gating 機構の判断軸: (i) routine 利用文脈で「user 判断委ね」path が累積すると stall surface が線形増加、(ii) safety-critical な structural note（cross-file 移動 / 大規模 rewrite 等）のみ surface 残置 / mechanical 適用可能なものは auto-apply、(iii) 「将来 caller が手動 review したい時」は YAGNI で trigger にしない。`/extract-rules --compact` v1.18.0 で `consolidation_proposals` を `caller-judgment` から `auto-apply via main-thread synthesis` に flip した実例。新規 routine skill 設計時は detection-only / auto-apply 二分を Decisions § N で明示判断
- **Subagent contract preservation via main-thread synthesis (detection-only contract + auto-apply via main-thread Edit synthesis)**: subagent contract が **detection-only / analysis-only / Forbidden tool calls** で確立されている Pattern A skill において auto-apply を実現する canonical pattern。subagent は構造化 output（例: `consolidation_proposals` の `cluster_bullets[].snippet` + `merged_principle.{name, text}` + `replacements[].{line_range, strategy, cross_ref_text}`）を emit のみ、**main thread が同 output から `Edit` calls を合成**して適用。subagent 側 SKILL.md / `references/<mode>-prompt.md` の "Forbidden tool calls" / "Materialization disposition" 句は **不変** に保ち（v1.17.0 same-day reversal signal を回避）、SKILL.md 本体に新 sub-phase（例: `(c2) main-thread synthesis (iter 1 only)`）を追加。利点: (i) subagent bias-free 維持、(ii) cluster-boundary drift が構造的に解消、(iii) Pattern A 2-layer 規律に一貫。auto-apply 化要求時の 1st candidate は「subagent 側 emit を flip」でなく「main thread synthesis layer 追加」
- **Pattern A iteration loop two-phase apply step (`(c1) mechanical_edits apply` + `(c2) main-thread synthesis`)** (class-level extension of Pattern A `(b) Parse + (c) Apply` canonical shape): Pattern A の `(c) Apply` phase が 2 種類 output array（apply 用 `mechanical_edits` + caller-judgment 用 `consolidation_proposals` / `structural_notes` 等）を持ち後者を auto-apply 化したい場合、`(c)` を `(c1) <apply-array> apply` + `(c2) <judgment-array> main-thread synthesis` に分割。必須: (i) `(c2)` は iter 1 only（`Iter-1-only capture discipline` rule との整合）、(ii) iter-level `applied_edits_count` counter を `(c1)`/`(c2)` 間で **共有**（convergence check が compaction-only / consolidation-only / mixed 経路で uniform）、(iii) `(c2)` 内クラスタは sequential 処理（cluster A complete before cluster B begins、cluster B が post-cluster-A 状態を Read）、(iv) `(c1)` の scope rail / no-op fallback 規律を `(c2)` も同 form で継承、(v) ambiguous-emit picker（subagent が同一 `line_range` に複数 strategy emit した場合）で `cross_ref` を `delete` より優先（incident pointer preservation）。`(c2)` の追加で `applied_edits_count > 0` semantics は「mechanical edit OR consolidation edit landed」に **widening** されるため `per_file_status` 判定 / CHANGELOG signal にも sweep が必要
- **Threshold buffer-ratio deliberate-design exception (zero buffer = warning-value match, with explicit rationale)** (specialization of `Threshold magic numbers anchored on observable platform signals + buffer ratio` canonical rule): canonical rule は 80% buffer を default としているが、user が「ワーニングが出るギリギリの値でもいい」と明示判断した場合は **zero buffer (warning-value-equivalent)** を許容。判断軸: (i) buffer の役割は「subsequent rule additions のための余白」で、現状運用で実害が無いなら zero buffer も intuitive design choice、(ii) Step 4 で Decisions § N に「default value」独立 question として並べ R / A の選択を user に委ねる（example: `36000 (90%)` / `38000 (95%)` / `40000 (warning と同値, buffer 0)` 三択）、(iii) user 既選択は CHANGELOG / Configuration description に「matches Claude Code's per-file warning threshold, observed in 2.1.x, with zero headroom by design」と明示。canonical rule との関係: Step 7.5 rules-review で「`80% buffer` rule 違反」が low-confidence finding として上がるが `Existing-rule literal-trigger exception` 3-point form（rule context = recommendation でなく manual user choice、本 PR context = explicit user gate 経由判断、目的 = subsequent addition 余白確保で実害無し）で reject。`Threshold magic numbers ...` rule を改定する別 PR が canonical fix の道、本 PR は rule-doc-drift signal のみ Step 11 へ route
- **Numeric-default-flip example-narrative self-consistency sweep** (class-level extension of `Coordinated multi-site sweep`'s multi-file-rename sub-case): numeric default 値を変更（例: `compaction_threshold: 32000 → 40000`）する task では、(i) **literal old value (`32000`) を全 file grep** で sweep（SKILL.md / Configuration table / Description / references 全 site）に加えて、(ii) **example narratives 内の illustrative numerics**（例: `report-templates.md` の `47600 → 35000 chars (over threshold)` で `47600 > 40000 > 35000` という大小関係を threshold value に依存して narrative 整合性を維持）も sweep target に明記。判断軸: (i) literal value grep は機械的 sweep で iter 1-2 catch されるが、(ii) example narrative の illustrative numerics は literal match しないため Code Review iter 3 まで漏れがち（実例: extract-rules v1.18.0 で `47600 → 35000` example が threshold 40000 と矛盾）、(iii) Plan の Test plan / Design 節に「example narrative の illustrative numerics 自己整合性」を sweep target として明示。canonical fix: (a) literal grep sweep + (b) `references/*.md` の例題で「閾値超過 / 閾値以下」例示数値を threshold に対する大小関係を保つよう調整、両方 Step 5 で 1 commit 内 sweep
- **Re-derive the full assessment, not just the raw value, when re-activating a task whose state was partially saved**: difficulty / enum 等の **assessment** から派生して N（反復数）や TodoWrite 行の completed 状態を pre-mark する設計で、task を再活性化（Step 4 rewrite-approach 等で plan を作り直す path）する際、**N と行状態だけ復元して assessment ラベルを据え置く partial restoration は stale 判定を残す**。下流の複数 gate が bare N でなく assessment 自体に keyed（例: `assessed Trivial` 条件で Step 4 unreviewed-plan 提示 / `references/plan-format.md` の reviewed-plan 文言条件 / Step 11.5 hard-skip が分岐）している場合、復元時に assessment を **再判定して全 derived 条件を一緒に re-derive** する。canonical: re-activation path に「N と TodoWrite 行だけでなく difficulty assessment 自体を再評価し、assessment に紐づく全下流 gate を同時に再導出する」と明記。判断軸は「下流 gate が raw value（N）でなく assessment ラベルを直接読むか」— 読むなら label の partial-restore が stale-condition 経路（実例: dev-workflow に Trivial 階層 N=0 を追加した際、Step 4 re-activation が N と行のみ復活させ Trivial ラベルが残ると下流 3 gate が stale な Trivial 判定に引きずられる Major finding が Step 8 code review iter 2 で surface）
- **Call-fail-then-fallback over availability-introspection for version-gated optional callees** (specialization of the sibling Prerequisites fallback bullets — reviewer / rules-review / extract-rules): optional に存在する callee skill（直近 upstream で再導入された built-in `simplify` 等、SKILL.md が on-disk に無く引数 interface も一次確認不能なもの）を primary、内製 skill を fallback にする resolution は、skill-list の **introspection で可用性判定しない**。canonical は **observable な呼び出し結果ベース**（まず `Skill(<primary>)` を呼び、skill-not-found 等で失敗したら retry once → `Skill(<fallback>)`）。判断軸: (i) introspection（tool-list 走査 / 可用性 query）は platform 挙動依存で fragile、observable call result は deterministic、(ii) 既存 sibling Prerequisites bullet（reviewer / rules-review / extract-rules の 3 fallback）と同形 phrasing で統一、(iii) primary 経路は **fallback 固有の引数を一切仮定しない**（interface 未確認のため `Base ref` 等を渡さず既定スコープ + best-effort な自然言語ヒントのみ）。**Auto-fallback の user-gate 要否**は substitution の等価性で分岐: functionally-equivalent な substitution（cleanup skill の simplify↔tidy 等、observable behavior が同じ）は **gate せず自動続行**（ただし `<primary> unavailable — falling back to <fallback>` 相当の 1 行 note を emit して沈黙にしない）、quality を materially 変える substitution（reviewer の差し替え等）は user-gate を挟む。resolution は Prerequisites bullet 1 箇所を source of truth とし Step 本体は cross-reference のみ（`Token defined-once + cross-reference` 整合）
- **Callee swap without phase/concept rename: distinct update-site vs preserve-site closed lists** (specialization of `Coordinated multi-site sweep`): SKILL.md の Step phase が呼ぶ callee skill を差し替える（例: Step 6 が `tidy` → `simplify` 優先に）が **phase 名・概念名は温存する**（fallback skill が残るため phase 名が orphan にならない / 意図的 distinguished name との衝突回避 / churn 抑制）場合、編集は 2 つの closed list に分けて sweep する: (i) **update-site = 呼ぶ skill を名指す箇所**（`allowed-tools` / dispatch / Agent-usage bullet / `§ No-Stall Principle` の callee 列挙 / Progress Visibility / 責務分担行）は primary/fallback 併記に更新、(ii) **preserve-site = phase / 概念を旧名で呼ぶ箇所**（`Step N: <PhaseName>` 見出し / TodoWrite 行 / phase-skip 例 / `<PhaseName>-revival check` / README の phase 表記）は不変。判断軸: callee-name reference と concept-name は別 sweep class で、両者を混同すると preserve すべき phase 名まで rename して不要な churn / fallback skill 名との不整合が発生。phase 名が fallback skill 名由来である旨を 1 文添える。canonical edit 後の bundle copy 同期 + `verify-bundle-sync` は通常どおり
- **Cross-environment platform-capability divergence → prefer-with-fallback, grant both in `allowed-tools` (not exclusive migration)**: 配布スキルが依存する platform capability（session task-management ツール等）が **環境ごとに利用可否が割れる**場合（例: 新ホスト版が新ツール群を default 化したが別の編集環境では旧ツールのみ露出）、新ツールへ exclusive 移行して旧ツールを `allowed-tools` から落とす設計は、旧ツールのみの環境で無条件破綻する。canonical は **prefer-with-fallback**: (i) 本文の主語は新ツールを基準にしつつ「新ツールが無い環境では同等の旧ツールを使う」フォールバック注記を 1 段落、(ii) `allowed-tools` に **新旧両ツールを併存** grant（エージェントが利用可能な方を選べる）、(iii) 両ツールの観測可能な semantics（status enum 等）が同一なら「全環境で動作・破綻なし」と CHANGELOG / commit body に明記。判断軸: 利用可否が environment-dependent な platform capability への移行は、exclusive 切替（最小権限・簡潔 prose を優先）か prefer-with-fallback（全環境動作を優先）かを Step 4 Decisions で並べる。配布物（広い環境で実行される bundle skill）では後者が default、単一環境前提の project-local なら前者も可。最小権限ルールとの整合は「cross-environment 要件による正当な例外」として rules-review の minimalism finding を reject
- **Fallback-path field-name bridge: enumerate field-name divergences between primary and fallback tool when prose references a primary-only field**: prefer-with-fallback 設計（上記 cross-environment 規律）で primary ツールと fallback ツールが **同一概念に異なる field 名**を持つ場合（例: 新ツールの記録欄が `description`、旧ツールでは `content`）、本文が primary 固有の field 名を名指す箇所はフォールバック経路で **存在しない欄への書き込みを誘発する guess-point** になる。canonical: (i) primary-only field 名を参照する prose には fallback 側の対応 field 名を括弧書きで併記（`the task's <primary-field> field (the <fallback-field> field under the <fallback-tool> fallback)`）、(ii) フォールバック注記導入時は **primary 固有 field 名参照を全 skill で grep** して class-level に bridge（write-path 欄名のみ — read-path は注記でブリッジ済みか、欄名参照が無い skill は対象外）。漏れは review 系フック（skill-review 等）が Major finding として catch。exclusive→fallback の approach 変更で新規に作り込みやすいギャップ
- **Reviewer "field does not exist" claim → verify against the loaded tool schema + the run's own successful calls, do not accept the assumption** (specialization of `Mis-justification audit when citing infrastructure mechanisms`): レビュア（peer / simplify / skill-review 等）が「この field は当該ツール固有ではない / 当該ツールには存在しない」型の false-positive を上げた場合（例: 「`activeForm` は TodoWrite 固有で Task* には無い」）、assumption をそのまま受け入れて field を削除しない。canonical 2-leg 一次確認: (i) **本セッションで in-context に loaded されている当該ツールの tool schema** に当該 field が（optional 含め）定義されているかを直接読む、(ii) **本走行自身の successful tool call**（例: Step 1 の `TaskCreate` が当該 field を渡して成功している）を live evidence として cross-check。両 leg が confirm すれば「共通 / 有効 field」と判定し no-op（削除しない）、その旨を「one finding が複数 review/simplify pass で再提起されても同じ一次情報で confirm 済」と記録して再 flag を抑制。判断軸: tool field の存否は LLM の事前知識（version 差・環境差で揺れる）でなく loaded schema + 実呼び出し結果という deterministic な一次情報で決める。`:27`（cited mechanism の実挙動 verify）の **tool-field-membership 版** specialization — そちらは設計制約の justify、こちらは reviewer false-positive の disposition
- **Verbatim-preserving prose reformat: closed-list anchor split + reverse-transform byte-equality verification**: delimiter 連結の run-on（例: ask-peer の Review-Focus セミコロン連結 bullet、約 15 句が単一行）を入れ子 bullet リスト等へ整形して可読性を上げるが **clause wording は verbatim 保持** する整形 task の canonical 2 規律。(i) **Closed-list anchor split**: top-level clause 境界でのみ split し、enumerated label の closed list を anchor にする。delimiter 全出現での naive split は禁止（parenthetical / sub-enumeration 内の同じ delimiter で over-fragment する）。(ii) **Reverse-transform byte-equality verification**: 整形が verbatim を壊していないかは visual diff でなく **逆変換 byte 一致** で検証する — 入れ子 bullet を元の delimiter（例: `"; "`）で rejoin し、整形前の run-on 行と byte-for-byte 比較する。bundle skill の場合 canonical / bundle copy 2 copy も byte-identical 維持。`simplify` / `tidy` の cleanup 観点（reuse / simplification / efficiency / altitude）は **コード対象**で、verbatim 不変条件下の prose では「簡素化」を禁止する旨を dispatch prompt に明示（agent が句を勝手に削る誤適用を防ぐ）。整形 diff は意味変更と混ぜず専用コミットに分離（sibling symmetry で他の同型 run-on も同コミットで整形可）

- **Decisions / clarifications surfaced only in chat prose are not findable at the approval gate — persist them into the plan document**: dev-workflow の Step 4 のような **approval gate が plan document を render する**設計では、ユーザーへの回答・読み合わせで合意した判断・clarification を **チャット本文だけに書いて済ませない**。承認モーダル / gate に映るのは plan 文書本体であり、チャット prose は ephemeral でそこに到達しない（実例: ユーザーから「その回答はチャットにしかなく後から見つけられない、プラン文書に書いて承認画面に出るようにせよ」と訂正された）。canonical: gate 通過に関わる決定・補足回答・読み合わせ結論は plan 文書（Decisions § N / Context / Approach 等の該当節）に書き込んでから gate を再提示する。判断軸は「この情報は承認モーダルに render される文書本体に載っているか、それともチャット履歴にしか無いか」— 後者なら findability gap で、gate 提示前に文書へ昇格する。`Plan rewrite triggered by user material change at Step 4 gate` rule の user-既選択記録規律と整合（あちらは rewrite 時の記録先、本 rule は「チャットのみ回答は文書へ昇格すべき」という findability の動機を明示）
- **Planned N-commit split collapses to one commit when the changes interleave within a single file (file-level-staging constraint)**: dev-workflow Step 10 のようにステージング粒度が **ファイル単位（hunk 分割不可）** の commit flow では、プランが概念上 N commit に分けるつもりでも（例: producer 側 / consumer 側を 2 commit に分割）、それらの変更が **同一ファイル内に混在**しているなら物理的に分離できず **1 commit へ統合**するのが正しい挙動。canonical: Step 10 の commit 提案時に「プランでは N commit を予定したが、変更が同一ファイル（`<path>`）に混在し本ワークフローのステージングはファイル単位のため 1 commit に統合」と 1 行 note を添えて提案する（無言で統合せず統合理由を明示）。判断軸は「分割したい論理境界が file boundary と一致するか」— 一致しないなら file-level staging の制約で統合が不可避。プランの commit 分割は **論理的意図**であって commit 数の契約ではない: file boundary をまたがない split は staged commit にできるが、同一ファイル内 split は統合する（PR 分割なら Step 1.5 の subtask 境界で別 PR にする道が別途ある）
