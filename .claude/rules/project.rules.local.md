# Project Rules - Local (Conversation-Extracted)

このファイルは `--from-conversation` で抽出された project-specific patterns を保持する `.local.md` です。`project.rules.md` の Principles と分離して、organizational rule sharing 時に portable な Principles のみを promote できるよう運用しています。

## Project-specific patterns

- **`jq` の `null` 文字列フォールバック**: `jq` は entry / key 不在時に literal `null\n` を **zero exit** で出すため `|| echo unknown` は効かない。canonical は filter 内 `// "unknown"` + post-pipeline `[ -z "$value" ] && value="unknown"` の 2 段構え（例: `jq -r '(.plugins[] | select(.name == "<n>") | .version) // "unknown"' .claude-plugin/marketplace.json 2>/dev/null`）。producer / consumer 両側が同 class なので class-level extension audit 対象
- **detached HEAD ガード**: branch 操作を伴う routine スキルでは Pre-flight に `git symbolic-ref -q HEAD >/dev/null` の non-zero exit を以って detached HEAD を検出し early abort。`git rev-parse --abbrev-ref HEAD` 単独だと literal `HEAD` を返し後段 cleanup で `HEAD` という名前のブランチを探して fail する
- **`refs/heads/<glob>` の single-quote**: SKILL.md prose で glob を含む refspec（`git for-each-ref ... refs/heads/triage-*`）は `'refs/heads/triage-*'` で single-quote する。zsh の `nomatch` がデフォルト有効でマッチ無しだと abort し routine が silent halt する。bash でも quote 推奨
- **Run-level invariant の Pre-flight への hoist**: per-Finding / per-iteration loop 内で同一値を毎回 resolve する設計（例: triage が毎 Finding `current_version` jq 解決 / CHANGELOG.md Read）は Step 1 Pre-flight に hoist してキャッシュ。N×M 累積解消 + error handling 集約。producer + consumer 両側を hoist
- **disposition enum 拡張ではなく既存 enum の厳格化**: 新 case を追加する際、新 disposition value 発明より **既存 criteria の厳格化** を優先（例: triage の "Already addressed in later version" は新 disposition でなく既存 reject 基準 #1 を 2-leg AND test で具体化）。enum 増加は downstream parser / mapping table 全 update + 後方互換性崩壊
- **dead flag よりも単一 counter で cleanup 条件を表現**: boolean flag + counter の合成条件は counter 単独で同等 semantics が表現できないか精査（triage の `triage_branch_active` は fatal abort path が Step 4 summary を経由しないため counter 単独で OK と判明）。Code Review で「dead flag drop」として上がる頻出 finding
- **Run-level 失敗系は per-Finding edge-case 表に混ぜない**: pre-flight 失敗（branch 作成失敗 / detached HEAD 等、loop 外）と per-Finding edge-case（loop 内）は semantics scope が違うため別節に分離。SKILL.md の Pre-flight 節で手続き的に記述し reference 側 edge-case 表には載せない
- **Step 6 Simplify の rationale 段落は配布スキルでも brevity を保つ**: subtle behavior の rationale は技術的に正確でも段落が膨らみがち（100 words 超）。iter 2 Code Review で冗長 finding として上がるため Simplify-revival check と並んで rationale prose brevity も意識する
- **Branch behavior は completion で specify する（negation 禁止）**: 「branch A と同じだが X を除く」型の negation specify は hidden gap を生む（後追い読み手が branch A defaults を正しく解釈する保証なし）。Code Review iter 1 で Critical 級頻出。canonical は (i) **Full <set-name> (applies to both <A> and <B>)** shared block を上に立て、(ii) branch-specific actions で差分のみ書く 2 階層。(E) error class shared updates も by-completion explicit list で書く。1 箇所 negation を発見したら同 SKILL.md 内全分岐を sweep（class-level extension audit）
- **Pre-implementation smoke test (Step 0) for undocumented platform capabilities**: 前例のない platform capability（`Skill()` を subagent から呼び出す / `subagent_type: general-purpose` の tool set / allowed-tools 継承等）に依存する architectural rewrite は Step 5 前に Step 0 smoke test phase を Test plan に組み込む。smoke test 必須項目: (i) feasibility PASS/FAIL、(ii) tool inheritance、(iii) state-management simulation、(iv) audit-trail (READY/NEEDS-FALLBACK/BLOCKED)。「推奨」でなく「必須 (Step 0 item #N)」と wording（推奨表記は iter 2 で Major flag）
- **Dispatch-layer health と callee-layer health の counter 分離**: subagent dispatch 経由で callee skills を呼ぶ設計で、(i) dispatch-layer error（subagent dispatch 失敗 / verdict parse / schema violation / tool error → `D_dispatch_error_count`）と (ii) callee-layer error（callee error verdict / max-iter → per-callee `<skill>_disabled`）は **counter を分離**。(E) dispatch-layer error path は per-callee disable counter を incremented も reset もしない（callee は走っていない）。Step 4 aggregate は 2 layer を別行で render し observability axis 直交を保つ
- **Sequential N-callee orchestration を 1 Agent dispatch に集約する architectural pattern**: N 個の sequential `Skill()` で orchestrator 側 N decision points の stall surface が積み上がる場合、1 個の `Agent` dispatch に集約して aggregate JSON 1 個返す設計に rewrite（stall surface N→1）。必須: (i) aggregate JSON schema が per-callee field を nest、(ii) (E) dispatch-layer error class を callee-layer から分離、(iii) `subagent_type` を Step 0 smoke test で確認、(iv) subagent prompt に「Do not run further `Skill()` dispatches beyond what is enumerated」で内部 stall surface も bound
- **Routing-field entry shape validation extension** (class-level extension of `Edit` old_string validation rule): entry shape validation は `Edit` の `old_string` だけでなく **downstream tool call が dereference する全フィールド**（例: `publicity_review.remaining_findings[].file` → `git checkout HEAD -- <entry>.file`）に拡張。parse 段階の schema-violation check で全 routing field を non-empty string と検査。`Edit`-focused のみだと iter 2 Code Review で Major finding
- **Bold-prose label cross-reference style** (specialization of stable heading reference rule): 参照先が `### Heading` でなく bold-prose label（`**Per-Finding record kept in memory**` 等）の場合、`§ <Heading>'s "<bold-prose label>" paragraph` 形式で bold 内文言を verbatim で囲み込む。Code Review で Major dangling cross-ref として上がりやすい。bold label 書き換え時は参照側も同期する Edit 時双方向 audit 対象
- **Symmetric / cross-file extension audit (sibling enum + SKILL.md ↔ `references/*.md`)**: 並列 sibling enum（`record.verify_diff` / `record.skill_review` / `record.publicity_review` 等）に新値追加時は全 sibling field に同一値を追加（asymmetric だと aggregate switch 文 / mapping table で取りこぼし）。**Cross-file 兄弟**: SKILL.md closed list ↔ `references/*-criteria.md` table 二重管理は (i) SKILL.md Edit 直後に reference table 行を同期する 2-step audit、(ii) pre-existing instance も sweep、(iii) reference 側に `Source of truth: SKILL.md § ...; keep in sync` directive を 1 行入れる。実例: triage の parse-error 新 case「Zero `### Finding` headings」を reference table に追加し忘れ Code Review iter 2 で Major finding。1 case 漏れ発見時は同 SKILL.md 内の他 closed list / sibling reference table 全 sweep
- **Plan rewrite triggered by user material change at Step 4 gate (+ swap specialization)**: Step 4 で user が Alternative 選択等を要求した場合、Step 3-(N+1) review iter を立ち上げてから ExitPlanMode 再提示。手順 (1 pass): (i) Title から旧 approach 語句削除、(ii) Context / Goal / Approach を新 approach に置換、(iii) Risks 節も新 approach 固有 unknowns に完全置換、(iv) Decisions § N に user 既選択を記録。**Swap-direction specialization** (user が Alternative 選択): (a) Recommendation / Alternative position swap、(b) 新 Recommendation 見出しに「user 既選択」annotation、(c) Risks / CHANGELOG note も新文脈に書き換え、(d) Approach / Scope / Test plan も同期更新。partial rewrite は plan 肥大化 + buried decisions / scope creep 頻出
- **Routine-side action ownership over environment-finalization delegation**: 非対話 routine が外部副作用（`git push` 等）を伴う場合、SKILL.md 内で「session finalization に委譲」と宣言しない（旧 triage の delegation 宣言は (i) session-level 「designated branch」instruction との誤読、(ii) local 環境 feature parity 欠落 を生んだ）。canonical: routine 自身が once per run で実行（単一箇所、cleanup 判定後 / summary 直前）、`allowed-tools` に必要 glob 追加、session-level rule との conflict 文脈と禁止条項を SKILL.md 内に明記
- **Closed form-set invariant for multi-form summary tokens**: SKILL.md summary line（例: `triage-branch:`）の render form 数は closed list として bounded（3 以下程度）。partial-failure / cleanup 中断は新 form でなく別 warning line（`cleanup: switch back failed (...)`）に分離。SKILL.md prose に「form set stays closed at <N>」と明示文書化して class-level extension audit 対象を明確化
- **External-command failure: retry once + stderr-derived `<reason>` + no auto-recovery**: 外部 CLI（`git push` / `gh api` / `curl` 等）失敗の canonical pattern は 3 点: (i) **retry 1 回のみ + 1–2 sec sleep**（指数バックオフは over-spec、deterministic failure は retry で改善しない）、(ii) **`<reason>` は stderr 最終 non-empty 行を ≤ 80 文字 truncate**、empty なら `(no stderr)` fallback、(iii) **auto-recovery 明示禁止**（`Do not auto-recover (no force push, no rebase, no branch rename)` — 禁止文言なしだと LLM が rebase 等を hallucinate）。retry rationale 段落は Simplify で narration 削除されるため prose に残さない
- **Mis-justification audit when citing infrastructure mechanisms**: 設計制約を別 mechanism の挙動で justify する場合、引用先の実挙動を必ず一次情報で確認（旧 triage は `for-each-ref` が origin を読むと暗示したが実際は local refs のみ — peer review iter 1 で Major finding）。canonical fix: (a) cited mechanism 実挙動 verify、(b) 誤りなら design intent ベースの justification に書き換える
- **Cross-section rendering-authority pointer (computed-here / rendered-there)**: 「section A で compute、section B で render」設計では両 section に明示的相互参照ポインタを置く（実行順 + 各 section の rendering 担当範囲が両側から読める）。実例: `§ Auto-cleanup` 冒頭に `Run before § Push triage branch to origin ...`、`§ Push triage branch to origin` 冒頭に `Run after the auto-cleanup decision settles ...`。「どの section が最終 render を確定するか」を曖昧にしない
- **Agent definition (`.claude/agents/<name>.md`) frontmatter `allowed-tools` requirement** (class-level extension of Pattern A `allowed-tools` baseline mirror): agent 定義ファイルも `allowed-tools` 宣言なしだと nested dispatch (`Skill()` / `Agent` / `Bash(...)`) が silent fail（SKILL.md と違ってテスト系で検出されにくく実走行で初 surface）。canonical: (i) dispatch する callee `Skill(<name>)` を明示、(ii) callee `allowed-tools` の union を列挙、(iii) `Bash(*)` を避け callee と同じ glob 粒度。新規 agent / callee 構成変更時は callee SKILL.md frontmatter を 1 行ずつ diff して union を書き写す 2-step audit
- **Subagent inline-execution prohibition for `Skill(<callee>)` dispatches in agent definitions**: agent `allowed-tools` に `Skill(<callee>)` を含めると callee SKILL.md が subagent context に injected され、subagent が dispatch せず callee logic を inline 再実行して内部 verdict JSON を直接返す failure mode（実例: triage-per-finding-reviewer が verify-diff Workflow を inline 実行 → aggregate でなく `{"status": "converged", ...}` を返し orchestrator (E.2) で 3 連続 fail → `D_dispatch_disabled` で残り全 Finding skip）。canonical mitigation: agent 定義に `## Dispatch discipline` 節を新設し 3 段落で reinforcement-by-repetition: (i) MUST 宣言「Each callee MUST be invoked via `Skill(<name>)`. Do not read, interpret, or replicate any callee's SKILL.md logic inline」、(ii) concrete restatement、(iii) closed-list bound「Do not run further `Skill()` dispatches beyond the N enumerated above」。3 段落重複は positive obligation / concrete / closed-set bound の 3 angle 意図的補強。anti-pattern: callee 内部スキーマの field 名を agent prose に書く（inline 実行の材料を増やす）
- **Routing-identifier permissive / content strict layering**: 「routing 用 identifier fields（title format / 識別子 regex 等）」と「judgment ロジックが参照する content fields（Finding 内 enum / required fields 等）」が併存する場合、**identifier は permissive、content は strict** に分離。両 layer 同水準 strict だと grammar 揺れ（`1 finding` vs `N findings` 単複、trailer 有無等）で silent skip regression。判断軸: (i) 事前フィルタ済み chamber なら identifier format check 不要、(ii) 保持時は permissive form（`\d+ finding(s)?`、trailer は optional cross-check）、(iii) content は strict 維持。design rule of thumb を SKILL.md prose に明文化。format check 廃止が大変更なら Step 4 user gate に出す
- **Vacuous-truth gap in per-element ALL-quantifier close predicates**: per-element ALL-quantifier predicate（`every Finding is OK to close` 等）は zero records 時に vacuously true となり parse-error / dispatch-aborted 等の whole-state を silently 同 branch に倒す（実例: triage の close decision が parse-error 空 Finding 配列で vacuous all-pass → auto-close 起動）。canonical fix は 2-leg AND predicate: (i) whole-state gate（closable whole-state — not parse-error, not aborted）、AND (ii) per-element check。SKILL.md prose に vacuous-truth 注記を 1 行入れる。Step 3 plan review は abstract 扱いで Step 8 まで漏れる Critical 頻出
- **Step renumbering propagation in SKILL.md (mechanical sweep + individual overrides)**: Step 番号 renumbering は Design 節で 2 階層記述: (i) **Generic rule (mechanical sweep)** — 単純置換可な参照群列挙、(ii) **Individual overrides** — sweep だと意味的誤りになる箇所を line 別列挙（代表的: 旧 `Runs after Step N` が新番号で当該 hook 自体が Step N になるケースは `Runs as Step N (immediately after Step N-1)` に書き換え、Step 8 reminder の `Step 9 transition` を `Step 9 (Completion Hooks) transition` で disambiguate）。**line 番号は pre-edit reference** 明記、最終確認は phrase / heading anchor。`references/*.md` 波及は § J で line-by-line checklist
- **TodoWrite single-row for unknown sub-iteration count at registration time**: sub-iteration count が user-approval input に依存する Step（Step 10 Interactive Commits の commit 数は approval gate 通過で確定等）は per-iteration row に展開せず single row 登録（理由: M 未確定 / mid-flight 書き換えは stall surface 増、per-commit 進捗は Step 内 prose で扱う）。Step 1 sub-step 7 に `(single row — per-X iteration is handled inline within Step N because the X count is not known until <ref>)` annotation。既存 Step 3 / Step 8 review iter は `review_iterations` で count 確定なので per-iter 展開 — 両 pattern を混同しない
- **Behavior-change default-flip → CHANGELOG opt-out note + downstream automation visibility**: default value flip は minor semver bump 単独だと signal 強度不足の可能性あり。CHANGELOG entry に 2 要素明示: (i) **opt-out 手順を冒頭で loud に表記** — `**Default: enabled** — set \`<flag>: <old-default>\` in \`<config-path>\` to opt out` を first sentence に、(ii) **downstream automation note** — Risks / CHANGELOG note で「自動 update する CI は CHANGELOG を読まずに走るため signal 補完が必要」を明示。Step 4 で default flip 判断した場合は Decisions rationale に直接組み込む
- **Multi-form gate vocabulary disambiguation in § No-Stall Principle enumeration**: 1 Step が複数 user-gate を持ち taxonomy が異なる場合（Step 10 の commit-plan approval / per-commit accept (4 値) / fold-or-defer (binary) / ambiguous-adjust clarifier 等）、§ No-Stall Principle で gate ごとに別 bullet 列挙（merge 禁止）。各 bullet は canonical token classifier の定義段落を `§ Step <N>'s "<paragraph label>" paragraph` で明示参照。conflate すると enum 取り違え（Code Review iter 3 Major gate count 不整合）
- **Filename-robust git collection + per-commit pathspec scoping for Step 10 / commit flows**: canonical form は `git status --porcelain=v1 --untracked-files=all -z` ((i) `=v1` で format pin、(ii) `--untracked-files=all` で user config override、(iii) `-z` で NUL-separated + C-quoting 抑制 — `-z` 省略は downstream `git add` で pathspec 不一致)。**Untracked Read fallback for "all-changes diff"**: `git diff <base>` は untracked を含まないため tracked = `git diff` portion、untracked = `Read` で取得し new-file hunk として present。`??` prefix を discriminator、Procedure 冒頭で once 実行して in-memory reuse。**Multi-pathspec single call**: `git add -- "<f1>" "<f2>" ...` 一括発行、`-A` / `git add .` は禁止（unrelated drift staging）。**Stop-hook auto-stages**: Web 環境の `~/.claude/stop-hook-git-check.sh` が untracked を auto-stage するため各 commit を `git commit -m <msg> -- <path-1> ...` で pathspec scope（flag order: `-m` を `--` より前）。作業 staging 文書は `.gitignore` でなく commit pathspec で scope-out（gitignore は team-shared plan を消す）
- **Counter lifecycle + counter-based routing (zero-exit-only increment + amend exclusion + actuality routing)**: 「成功した landing 単位 counter」(`landed_count` / `triage_commit_count` 等) の canonical lifecycle は 4 点 explicit: (i) loop 開始時 0 init、(ii) zero exit 時のみ +1、(iii) retry / abort / failed path は no increment、(iv) `--amend` / re-commit は再 increment しない（initial landing で既にカウント済み）。SKILL.md に「On zero exit, increment by 1」「The amend ... is NOT re-incremented」symmetric 明示。**Downstream routing は config-flag でなく counter で判定**: `landed_count > 0` で routing（config-flag の intent で routing すると `enabled: true` × skip 条件で 0 commits 終了 case を取り違える）。「`landed_count > 0`: ... `landed_count == 0` (either because `<flag>: false` or because <Step> was skipped): ...」と 2 cause を OR union で書く
- **2-stage grep audit for cross-file Step renumbering**: Step 番号 renumbering 最終確認は 2 段階 grep: (1) **phrase-pair-only grep**（旧 section heading 句 OR）で 0 hit 確認、(2) **word-boundary grep on bare numbers**（`\b(Step 9|Step 9\.5|Step 10)\b`）で残り hit 目視。phrase-only は bare 漏れ、bare-only は noise 大、2 段構成で false-neg / false-pos 両抑え。Test plan / Design § J の line-by-line checklist と組み合わせ、**line 番号は pre-edit reference** 明記
- **Token defined-once + cross-reference (re-render 禁止)**: 複数箇所から参照される token（localized / status / enum mapping）は 1 箇所定義 + 他箇所は cross-reference（re-render すると canonical 更新が drift する、i18n token は特に高頻度 wording 更新）。cross-ref 構文は既存ルール `Bold-prose label cross-reference style` の `§ <Heading>'s "<bold label>" paragraph` 形式。Simplify で re-render finding 出たら Step 6 で 1 source-of-truth に統合
- **Temporary-workaround skill integration: single hook point over state-machine weave**: 暫定対処として追加する project-local skill（`anthropics/claude-code#53948` symlink bug の workaround 等）は既存 routine skill に組み込む際 **state machine を貫通せず単一 hook point に閉じ込める**: (i) outer state machine / Per-Finding record schema / counter / TodoWrite rows / No-Stall reminder enumeration に一切触れない、(ii) 既存 sub-step 境界に 1 段落の sub-step 挿入（例: `(f.5)`）、(iii) deletion 手順を localization rationale として SKILL.md に明記（closed list で）。Step 1 difficulty assessment で「暫定 = 削除予定」と判明した時点で deep integration を early reject、単一 hook point を Recommendation に立てる
- **Edit-induced false-positive: widen edit scope, not add detect-only check**: verification skill が持続的 fail する場合、root cause は多くが **upstream edit operation の scope incomplete**。fix 方向は (i) ❌ verification 側 fail-open、(ii) ❌ detect-only + conflict cascade、(iii) ✅ **edit 側 scope を広げる**（例: canonical edit 直後に `cp -R` で bundle copy へ sync する phase 追加、phase 1 sync → phase 2 verify の 2 段構成）。auto-apply で fail-open に倒すと structural 漏れ隠蔽になるので Step 8 user-gate に出す
- **Project-local skill placement and layout**: marketplace 配布対象でない skill（temporary workaround / project 固有 routine）は `.claude/skills/<name>/SKILL.md` **flat layout** + `marketplace.json` 未登録。bundle skill の nested layout (`skills/<name>/skills/<name>/`) と混同しない。判定軸: (i) 配布性必要 → nested + marketplace.json 登録 + CHANGELOG ペア bump、(ii) project-local → flat + 登録なし + CHANGELOG 対象外。誤配置すると symlink / marketplace.json entry / CHANGELOG / deprecation pointer の 4 箇所修正が必要
- **Pre-existing layout flag vs new-change regression discrimination**: test が TEST_FAILED を返した時、(a) 自分の change が導入した regression か (b) base-commit 時点で既存の pre-existing state かを必ず discriminate。check: (i) `git stash` or base commit clone で同 test 再実行、(ii) 同 failures なら pre-existing、(iii) test agent が flag した plugin/skill 名と change set file path を grep 交差確認、(iv) Step 8 では「pre-existing failures (out of scope of this PR)」と明記。pre-existing を PR で修正しようとすると scope creep
- **Stall mitigation: callee-side wording (A) + orchestrator-side pre/return-point reminders (B) — 2-stage serial pair**: stall 解消は (A)+(B) 両セット必要（片方だけは partial fix、(A) なしの (B) は callee prompt-injection に相殺される）。**(A) Callee-side schema-verb wording**: callee の fenced JSON contract に terminal-action 動詞（`End every invocation` / `Terminate` / `Do not produce any additional turn` 等）を使うと orchestrator が「turn を閉じろ」指示として受け取り stall。canonical: (i) schema-verb form「`Emit a single fenced JSON block at the end of the response, matching the schema ...:`」(`End` / `Terminate` 等は禁止)、(ii) 独立 `## Sub-skill caller directive` section に「the fenced JSON verdict block this skill emits is the structured return value of the skill's procedure — it is not a deliverable to the user, and emitting it does not terminate the orchestrator's turn」+「The same agent that ran this skill must immediately issue the next tool call dictated by the orchestrator's flow」+「Do not insert a prose summary, an acknowledgment, or a "shall I proceed?" sentence between the JSON verdict and the next tool call」、(iii) uniqueness clause「Only one fenced JSON block — the verdict block — appears in the response」。3 callee 配置時は byte-identical wording（token のみ差し替え）。新規 / 編集時は terminal-action verb を必ず grep。**(B) Orchestrator-side entry + return-point reminder (orthogonal coverage)**: return-point だけでは「決定の瞬間」を救えない（callee prompt-injection で turn 既閉時に読まれる）。canonical: **entry-side `Pre-invocation reminder`** を dispatch **直前**に挿入し return-point と直交化。構造: (i) next tool call の named specification、(ii) JSON verdict as return value (not turn boundary) framing、(iii) `See § No-Stall Principle.`。duplication は意図的 reinforcement-by-repetition で Simplify-revival check で削減しない旨を 1 行 rationale で残す
