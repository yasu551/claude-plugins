# Project Rules - Local (Conversation-Extracted)

このファイルは `--from-conversation` で抽出された project-specific patterns を保持する `.local.md` です。`project.rules.md` の Principles と分離して、organizational rule sharing 時に portable な Principles のみを promote できるよう運用しています。

## Project-specific patterns

- **`jq` の `null` 文字列フォールバック**: `jq` は entry / key 不在時 literal `null\n` を **zero exit** で出すため `|| echo unknown` は効かない。canonical は filter 内 `// "unknown"` + post-pipeline `[ -z "$value" ] && value="unknown"` 2 段構え（例: `jq -r '(.plugins[] | select(.name == "<n>") | .version) // "unknown"' file 2>/dev/null`）。producer / consumer 両側が同 class なので class-level extension audit 対象
- **detached HEAD ガード**: branch 操作を伴う routine では Pre-flight に `git symbolic-ref -q HEAD >/dev/null` non-zero exit で detached HEAD 検出し early abort。`git rev-parse --abbrev-ref HEAD` 単独だと literal `HEAD` を返し cleanup で `HEAD` という名前のブランチを探して fail
- **`refs/heads/<glob>` の single-quote**: glob を含む refspec（`git for-each-ref ... refs/heads/triage-*`）は `'refs/heads/triage-*'` で single-quote。zsh の `nomatch` がマッチ無しで abort し routine が silent halt する。bash でも quote 推奨
- **Run-level invariant の Pre-flight への hoist**: per-Finding / per-iteration loop 内で同一値を毎回 resolve（例: triage が毎 Finding `current_version` jq 解決 / CHANGELOG.md Read）は Step 1 Pre-flight に hoist してキャッシュ。N×M 累積解消 + error handling 集約。producer + consumer 両側を hoist
- **disposition enum 拡張ではなく既存 enum の厳格化**: 新 case 追加時、新 disposition value 発明より **既存 criteria の厳格化** を優先（例: triage の "Already addressed in later version" は新 disposition でなく既存 reject 基準 #1 を 2-leg AND test で具体化）。enum 増加は downstream parser / mapping table 全 update + 後方互換性崩壊
- **dead flag よりも単一 counter で cleanup 条件を表現**: boolean flag + counter の合成は counter 単独で同等 semantics 表現可能か精査（triage の `triage_branch_active` は fatal abort path が Step 4 summary を経由しないため counter 単独 OK）。Code Review で「dead flag drop」頻出 finding
- **Run-level 失敗系は per-Finding edge-case 表に混ぜない**: pre-flight 失敗（branch 作成失敗 / detached HEAD、loop 外）と per-Finding edge-case（loop 内）は semantics scope が違うため別節分離。SKILL.md Pre-flight で手続き的に記述、reference 側 edge-case 表には載せない
- **Step 6 Simplify の rationale 段落は配布スキルでも brevity を保つ**: subtle behavior rationale は技術的に正確でも段落が膨らみがち（100 words 超）。iter 2 Code Review で冗長 finding として上がるため Simplify-revival check と並んで rationale prose brevity も意識する
- **Branch behavior は completion で specify する（negation 禁止）**: 「branch A と同じだが X を除く」型 negation は hidden gap（branch A defaults 解釈保証なし）。Code Review iter 1 Critical 頻出。canonical 2 階層: (i) **Full <set-name> (applies to both <A> and <B>)** shared block、(ii) branch-specific actions で差分のみ。error class shared updates も by-completion explicit list。1 箇所 negation 発見で同 SKILL.md 内全分岐 sweep（class-level extension audit）
- **Pre-implementation smoke test (Step 0) for undocumented platform capabilities**: 前例のない platform capability（`Skill()` を subagent から呼び出す / `subagent_type: general-purpose` tool set / allowed-tools 継承等）依存の architectural rewrite は Step 5 前に Step 0 smoke test phase を Test plan に組み込む。必須項目: (i) feasibility PASS/FAIL、(ii) tool inheritance、(iii) state-management simulation、(iv) audit-trail (READY/NEEDS-FALLBACK/BLOCKED)。「必須 (Step 0 item #N)」wording（「推奨」は iter 2 で Major flag）
- **Dispatch-layer health と callee-layer health の counter 分離**: subagent dispatch 経由で callee skills を呼ぶ設計で (i) dispatch-layer error（dispatch 失敗 / verdict parse / schema violation / tool error → `D_dispatch_error_count`）と (ii) callee-layer error（callee error verdict / max-iter → per-callee `<skill>_disabled`）は **counter 分離**。dispatch-layer error path は per-callee disable counter を increment / reset しない（callee は走っていない）。Step 4 aggregate は 2 layer 別行で render し observability axis 直交を保つ
- **Sequential N-callee orchestration を 1 Agent dispatch に集約する architectural pattern**: N 個 sequential `Skill()` で N decision points stall surface が積む場合、1 `Agent` dispatch に集約し aggregate JSON 返す（stall surface N→1）。必須: (i) aggregate JSON schema が per-callee field nest、(ii) dispatch-layer error class を callee-layer から分離、(iii) `subagent_type` を Step 0 smoke test 確認、(iv) subagent prompt に「Do not run further `Skill()` dispatches beyond what is enumerated」で内部 stall surface も bound
- **Routing-field entry shape validation extension** (class-level extension of `Edit` old_string validation rule): entry shape validation は `Edit` の `old_string` だけでなく **downstream tool call が dereference する全フィールド**（例: `publicity_review.remaining_findings[].file` → `git checkout HEAD -- <entry>.file`）に拡張。parse 段階の schema-violation check で全 routing field を non-empty string 検査。`Edit`-focused のみだと iter 2 Code Review で Major finding
- **Bold-prose label cross-reference style** (specialization of stable heading reference rule): 参照先が `### Heading` でなく bold-prose label（`**Per-Finding record kept in memory**`）の場合、`§ <Heading>'s "<bold-prose label>" paragraph` 形式で bold 内文言 verbatim 囲み込み。Code Review で Major dangling cross-ref として上がる。bold label 書き換え時は参照側も同期する Edit 時双方向 audit 対象
- **Coordinated multi-site sweep on extension/addition**: 新しい値・member・gate・rename token 追加は coordinated site 全てを closed list で enumerate し 1 commit 内 sweep。canonical 形: (i) trigger 明示、(ii) 同期 site 列挙（sub-case 詳細は後続 bullet）、(iii) 1 commit atomicity、(iv) Edit 直後の同期 audit (pre-existing instance も sweep)、(v) reference 側に `Source of truth: ...; keep in sync` directive。漏れは Step 8 / 9 reviewer iter で cross-file inconsistency catch
- See pattern: Coordinated multi-site sweep (sibling enum 値追加: 全 sibling 同一値 + SKILL.md closed list ↔ `references/*-criteria.md` table 二重管理を 2-step audit + pre-existing sweep + `Source of truth` directive。実例: triage parse-error「Zero `### Finding` headings」追加忘れ)
- **Plan rewrite triggered by user material change at Step 4 gate (+ swap / insertion specializations)**: Step 4 で user が Alternative 選択等要求時、Step 3-(N+1) review iter 後 ExitPlanMode 再提示。手順 1 pass: Title から旧 approach 語句削除、Context / Goal / Approach 置換、Risks 完全置換、Decisions § N に user 既選択記録。**Swap-direction** (Alternative 選択): R / A position swap + 新 R に「user 既選択」annotation + Risks / CHANGELOG note / Approach / Scope / Test plan sweep。**Insertion-direction** (Decisions 不在の choice — 典型: default value): swap でなく **新 Decision § (N+1) を insert**、見出しに `**user 既選択: <value> — Step 4 gate にて**` annotation、R = user 選択 / A = 元暗黙 default、Approach / Scope / Test plan / Risks / CHANGELOG を新 default 文脈 sweep（旧 default 固有 Risks 完全置換）。swap と insertion の混同は元 structure 壊す。partial rewrite は plan 肥大化 + buried decisions / scope creep 頻出
- **Routine-side action ownership over environment-finalization delegation**: 非対話 routine が外部副作用（`git push` 等）を伴う場合、「session finalization に委譲」と宣言しない（session-level instruction との誤読 + local 環境 feature parity 欠落）。canonical: routine 自身が once per run で実行（単一箇所、cleanup 判定後 / summary 直前）、`allowed-tools` に glob 追加、session-level rule との conflict 文脈と禁止条項を明記
- **Closed form-set invariant for multi-form summary tokens**: SKILL.md summary line（例: `triage-branch:`）の render form 数は closed list として bounded（3 以下程度）。partial-failure / cleanup 中断は新 form でなく別 warning line（`cleanup: switch back failed (...)`）に分離。SKILL.md prose に「form set stays closed at <N>」明示文書化し class-level extension audit 対象を明確化
- **External-command failure: retry once + stderr-derived `<reason>` + no auto-recovery**: 外部 CLI（`git push` / `gh api` / `curl`）失敗の canonical pattern 3 点: (i) **retry 1 回のみ + 1–2 sec sleep**（指数バックオフ over-spec、deterministic failure は retry で改善しない）、(ii) **`<reason>` は stderr 最終 non-empty 行 ≤ 80 文字 truncate**、empty なら `(no stderr)`、(iii) **auto-recovery 明示禁止**（`Do not auto-recover (no force push, no rebase, no branch rename)` — 禁止文言なしだと LLM が rebase を hallucinate）。retry rationale 段落は Simplify で削除されるため prose に残さない
- **Mis-justification audit when citing infrastructure mechanisms**: 設計制約を別 mechanism の挙動で justify する場合、引用先の実挙動を一次情報で確認（旧 triage は `for-each-ref` が origin を読むと暗示したが実際は local refs のみ — peer review iter 1 Major finding）。canonical fix: (a) cited mechanism 実挙動 verify、(b) 誤りなら design intent ベースに書き換え
- **Cross-section rendering-authority pointer (computed-here / rendered-there)**: 「section A で compute、section B で render」設計では両 section に明示的相互参照ポインタ（実行順 + rendering 担当範囲が両側から読める）。実例: `§ Auto-cleanup` 冒頭に `Run before § Push triage branch to origin ...`、`§ Push ...` 冒頭に `Run after the auto-cleanup decision settles ...`。「どの section が最終 render を確定するか」を曖昧にしない
- **Agent definition (`.claude/agents/<name>.md`) discipline (2-clause)**: (i) **Frontmatter `allowed-tools` requirement** (class-level extension of Pattern A baseline mirror) — 宣言なしだと nested dispatch (`Skill()` / `Agent` / `Bash(...)`) が silent fail（実走行で初 surface）。canonical: callee `Skill(<name>)` 明示、callee `allowed-tools` union 列挙、`Bash(*)` 回避。(ii) **Subagent inline-execution prohibition** — agent `allowed-tools` に `Skill(<callee>)` を含めると callee SKILL.md が subagent context に injected され、subagent が dispatch せず inline 再実行する failure mode（実例: triage-per-finding-reviewer が verify-diff を inline 実行 → `{"status": "converged"}` を返し (E.2) で 3 連続 fail → `D_dispatch_disabled` で残り全 Finding skip）。canonical mitigation: agent 定義に `## Dispatch discipline` 節新設し 3 段落 reinforcement-by-repetition: (i) MUST 宣言「Each callee MUST be invoked via `Skill(<name>)`. Do not read, interpret, or replicate any callee's SKILL.md logic inline」、(ii) concrete restatement、(iii) closed-list bound。anti-pattern: callee 内部スキーマの field 名を agent prose に書く
- **Routing-identifier permissive / content strict layering**: 「routing identifier fields（title format / regex）」と「judgment 用 content fields（Finding 内 enum / required fields）」併存時 **identifier permissive / content strict** 分離。両 layer strict だと grammar 揺れ（`1 finding` vs `N findings`、trailer 有無）で silent skip regression。判断軸: 事前フィルタ済 chamber なら format check 不要、保持時は permissive form（`\d+ finding(s)?`）、content strict 維持
- **Vacuous-truth gap in per-element ALL-quantifier close predicates**: per-element ALL-quantifier（`every Finding is OK to close`）は zero records で vacuously true、parse-error / dispatch-aborted whole-state を silently 同 branch に倒す（実例: triage close decision が parse-error 空 Finding で vacuous all-pass → auto-close 起動）。canonical 2-leg AND predicate: (i) whole-state gate（not parse-error, not aborted）AND (ii) per-element check。Step 3 abstract 扱い → Step 8 漏れ Critical 頻出
- **Step renumbering propagation in SKILL.md (mechanical sweep + individual overrides + 2-stage grep audit)**: Design 2 階層: (i) **Generic rule** 単純置換群列挙、(ii) **Individual overrides** 意味的誤りになる箇所を line 別列挙（代表: `Runs after Step N` が新番号で hook 自体が Step N になるケースは `Runs as Step N (immediately after Step N-1)`、`Step 9 transition` を `Step 9 (Completion Hooks) transition` で disambiguate）。line 番号は pre-edit reference、最終確認は phrase / heading anchor。`references/*.md` 波及は § J で line-by-line checklist。**最終 audit 2 段階 grep**: (1) phrase-pair-only grep（旧 heading 句 OR）で 0 hit、(2) word-boundary grep on bare numbers（`\b(Step 9|Step 9\.5|Step 10)\b`）で残り hit 目視
- **TodoWrite single-row for unknown sub-iteration count at registration time**: sub-iteration count が user-approval 依存の Step（Step 10 Interactive Commits の commit 数は approval gate 通過で確定）は per-iteration row 展開せず single row 登録（M 未確定 + mid-flight 書き換えは stall surface 増）。Step 1 sub-step 7 に `(single row — per-X iteration is handled inline within Step N because the X count is not known until <ref>)` annotation。Step 3 / Step 8 review iter は `review_iterations` で count 確定なので per-iter 展開 — 両 pattern 混同しない
- **Behavior-change default-flip → CHANGELOG opt-out note + downstream automation visibility**: default flip は minor semver bump 単独だと signal 不足。CHANGELOG entry 2 要素: (i) **opt-out 手順を冒頭で loud** — `**Default: enabled** — set \`<flag>: <old-default>\` in \`<config-path>\` to opt out` を first sentence、(ii) **downstream automation note** — Risks / CHANGELOG note に「自動 update CI は CHANGELOG を読まず走るため signal 補完必要」。Step 4 default flip 判断時は Decisions rationale 組み込み
- **Multi-form gate vocabulary disambiguation in § No-Stall Principle enumeration**: 1 Step が複数 user-gate を持ち taxonomy 異なる場合（Step 10 commit-plan approval / per-commit accept (4 値) / fold-or-defer (binary) / ambiguous-adjust clarifier）、gate ごとに別 bullet 列挙（merge 禁止）。各 bullet は canonical token classifier 定義段落を `§ Step <N>'s "<paragraph label>" paragraph` で参照。conflate で enum 取り違え
- **Filename-robust git collection + per-commit pathspec scoping for Step 10 / commit flows**: canonical form `git status --porcelain=v1 --untracked-files=all -z`（`=v1` format pin / user config override / `-z` NUL-separated + C-quoting 抑制）。**Untracked Read fallback**: `git diff <base>` は untracked 含まないため tracked = `git diff`、untracked = `Read` で new-file hunk、`??` prefix を discriminator、Procedure 冒頭で once 実行 in-memory reuse。**Multi-pathspec single call**: `git add -- "<f1>" "<f2>" ...` 一括、`-A` / `git add .` 禁止（unrelated drift staging）。**Stop-hook auto-stages**: Web 環境の `~/.claude/stop-hook-git-check.sh` が untracked auto-stage するため各 commit を `git commit -m <msg> -- <path-1> ...` で pathspec scope（`-m` を `--` より前）。staging 文書は `.gitignore` でなく commit pathspec で scope-out（gitignore は team-shared plan 消す）
- **Counter lifecycle + counter-based routing (zero-exit-only increment + amend exclusion + actuality routing)**: counter (`landed_count` / `triage_commit_count`) lifecycle 4 点: loop 開始 0 init、zero exit 時のみ +1、retry / abort / failed path no increment、`--amend` / re-commit は再 increment しない。SKILL.md に「On zero exit, increment by 1」「The amend ... is NOT re-incremented」symmetric 明示。**Downstream routing は config-flag でなく counter**: `landed_count > 0` で routing（config-flag intent だと `enabled: true` × skip 条件の 0 commits case を取り違える）。「`landed_count > 0`: ... `landed_count == 0` (either because `<flag>: false` or because <Step> was skipped): ...」と 2 cause OR union
- **Token defined-once + cross-reference (re-render 禁止)**: 複数箇所参照 token（localized / status / enum mapping）は 1 箇所定義 + 他箇所は cross-reference（re-render で canonical 更新 drift、i18n token は高頻度 wording 更新）。cross-ref 構文は `Bold-prose label cross-reference style` の `§ <Heading>'s "<bold label>" paragraph` 形式。Simplify で re-render finding 出たら Step 6 で 1 source-of-truth に統合
- **Temporary-workaround skill integration: single hook point over state-machine weave**: 暫定対処の project-local skill（`anthropics/claude-code#53948` symlink bug 等）を既存 routine skill に組み込む際 **state machine 貫通せず単一 hook point に閉じ込める**: (i) outer state machine / record schema / counter / TodoWrite rows / No-Stall reminder enumeration に触れない、(ii) 既存 sub-step 境界に 1 段落挿入（例: `(f.5)`）、(iii) deletion 手順を SKILL.md に明記。Step 1 で「暫定 = 削除予定」判明時点で deep integration を early reject、単一 hook point を Recommendation に立てる
- **Edit-induced false-positive: widen edit scope, not add detect-only check**: verification skill が持続 fail 時、root cause は多くが **upstream edit operation の scope incomplete**。fix 方向: (i) ❌ verification 側 fail-open、(ii) ❌ detect-only + conflict cascade、(iii) ✅ **edit 側 scope を広げる**（例: canonical edit 直後に `cp -R` で bundle copy sync phase 追加、phase 1 sync → phase 2 verify の 2 段構成）。auto-apply で fail-open に倒すと structural 漏れ隠蔽 → Step 8 user-gate
- **Project-local skill placement, layout, and self-description**: marketplace 配布対象でない skill は `.claude/skills/<name>/SKILL.md` **flat layout** + `marketplace.json` 未登録（bundle skill の nested layout と混同しない）。判定軸: 配布性必要 → nested + 登録 + CHANGELOG ペア bump、project-local → flat + 登録なし + CHANGELOG 対象外。誤配置で symlink / marketplace.json entry / CHANGELOG / deprecation pointer の 4 箇所修正。**Self-description canonical phrasing**: sibling と同一 phrasing — canonical form ``**project-local** skill (lives under `.claude/skills/<name>/`, not registered in `.claude-plugin/marketplace.json`)``。既存 sibling (`triage-review` / `verify-bundle-sync` / `verify-diff`) を `grep` で確認し punctuation / 語順まで mirror（prefix prose 不要 — fragment 形）
- **Globally-exposed project-local skill: relative symlink under `~/.claude/skills/<name>`**: repo 外環境からも呼び出したい場合 `~/.claude/skills/<name>` から repo 内 `.claude/skills/<name>/` へ symlink。**absolute path 禁止 / relative path 必須**（repo `mv` / rename で absolute は dangling）。canonical form は既存 `~/.claude/skills/*` convention に揃え `../../<rest-of-path>` 形（例: `ln -s ../../Sources/github.com/<owner>/<repo>/.claude/skills/<name> ~/.claude/skills/<name>`）。新規追加時は既存 `~/.claude/skills/` を `ls -la` で確認。SKILL.md 内で absolute path を例示しない（コピペで absolute 形が増殖）
- **`Read` / `Write` / Setup-mode safety (3-clause)**: (i) **Drop `test -f` / `test -d` pre-checks (TOCTOU anti-pattern)** — `Read` 自体が missing-file error を明示するため pre-check は冗長 + TOCTOU。canonical: pre-check 削除し `Read` error を停止条件、`allowed-tools` から `Bash(test *)` drop。例外: missing と other error を distinguish する用途なら残し SKILL.md に意図 1 行明記。(ii) **Setup-mode pre-existence re-check (defensive race guard)** — config 不在検出後 Setup-mode template `Write` 直前に **再度 `Read` で不在再確認**（race で既存 file clobber 防止）。canonical「Before writing: re-confirm `.claude/<config>.local.md` does not exist ... If it now exists, abort Setup mode and report the conflict — never overwrite」。Code Review Major 頻出。(iii) **`Read` error semantics: default to stop-with-error** — error が missing vs permission / encoding を reliably distinguish できない場合、Setup-mode 分岐は判別可能時のみ、判別不能なら stop-with-error fallback（誤分岐は既存 config clobber 経路）
- **Pre-existing layout flag vs new-change regression discrimination**: TEST_FAILED 時、(a) change が導入した regression か (b) base-commit pre-existing state か必ず discriminate。check: (i) `git stash` or base clone で同 test 再実行、(ii) 同 failures なら pre-existing、(iii) test agent が flag した plugin/skill 名と change set path を grep 交差確認、(iv) Step 8 では「pre-existing failures (out of scope of this PR)」と明記。pre-existing 修正は scope creep
- **Stall mitigation: callee-side wording (A) + orchestrator-side pre/return-point reminders (B) — 2-stage serial pair**: stall 解消は (A)+(B) 両セット必要（(A) なしの (B) は callee prompt-injection に相殺）。**(A) Callee-side schema-verb wording**: terminal-action 動詞（`End every invocation` / `Terminate` / `Do not produce any additional turn`）は orchestrator に「turn を閉じろ」と受け取られ stall。canonical: (i) schema-verb form「`Emit a single fenced JSON block at the end of the response, matching the schema ...:`」(terminal-action 禁止)、(ii) 独立 `## Sub-skill caller directive` 節に 3 段落「fenced JSON は **return value** で turn 境界ではない / 次 tool call を同 agent が immediately issue / JSON と次 tool call 間に prose / 確認 / 「shall I proceed?」挿入しない」、(iii) uniqueness clause「Only one fenced JSON block」。3 callee 配置時は byte-identical wording。新規 / 編集時は terminal-action verb を grep。**(B) Orchestrator-side entry + return-point reminder**: return-point だけでは「決定の瞬間」を救えない。canonical: **entry-side `Pre-invocation reminder`** を dispatch 直前に挿入し return-point と直交化。構造: (i) next tool call の named specification、(ii) JSON verdict as return value framing、(iii) `See § No-Stall Principle.`。duplication は意図的 reinforcement-by-repetition で Simplify-revival check で削減しない rationale を 1 行残す
