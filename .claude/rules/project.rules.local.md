# Project Rules - Local (Conversation-Extracted)

このファイルは `--from-conversation` で抽出された project-specific patterns を保持する `.local.md` です。`project.rules.md` の Principles と分離して、organizational rule sharing 時に portable な Principles のみを promote できるよう運用しています。

## Project-specific patterns

- **`jq` の `null` 文字列フォールバック**: 配布スキル内 SKILL.md / references で `marketplace.json` 等から optional フィールドを取り出す際、`jq -r '.x | select(...) | .field' file 2>/dev/null || echo unknown` 形式は **不十分**。`jq` は entry / key 不在時に literal `null\n` を stdout に出して **zero exit** するため、`|| echo unknown` がフォーカルしない。canonical pattern は `value=$(jq -r '(.plugins[] | select(.name == "<n>") | .version) // "unknown"' .claude-plugin/marketplace.json 2>/dev/null)` で **filter 内** の `// "unknown"` を使い、追加で `[ -z "$value" ] && value="unknown"` の post-pipeline `-z` ガードでファイル不在 / `jq` 自体の non-zero exit / 空出力を吸収する 2 段構えにする。producer 側 (issue 本文への version 埋め込み) と consumer 側 (triage の `current_version` 解決) の両方が同一 class なので、片方修正したら必ずもう一方も同 pattern で修正する（class-level extension audit の対象）
- **detached HEAD ガード**: 配布スキルの SKILL.md prose で「現在のブランチを保存してから `git switch -c` で新ブランチに切り替える」フローを指示する場合、`git rev-parse --abbrev-ref HEAD` 単独だと detached HEAD で `HEAD` という literal を返し、後段の `git switch <original_branch>` cleanup が `HEAD` という名前のブランチを探して fail する。Step 1 Pre-flight 等で `git symbolic-ref -q HEAD >/dev/null` の non-zero exit を以って detached HEAD を検出し、early abort で `detached HEAD: cannot save original branch reference` 等の理由を summary に出す。ブランチ操作を伴う routine スキル全般に同じ guard を入れる
- **`refs/heads/<glob>` の single-quote**: SKILL.md prose で `git for-each-ref --sort=-refname refs/heads/triage-* --format='%(refname:short)'` のように glob を含む refspec を書く場合、shell 展開を防ぐため `'refs/heads/triage-*'` で single-quote する。zsh の `nomatch` option がデフォルト有効で、マッチが無いと shell が abort する（routine 走行で silent halting の元）。bash でも quote しておく方が安全。`git for-each-ref` 系の命令を SKILL.md に書く際の必須慣行
- **Run-level invariant の Pre-flight への hoist**: per-Finding / per-iteration loop の中で同一値を毎回 resolve する設計（例: triage の Step 3.3 で marketplace.json から `current_version` を毎 Finding 解決、CHANGELOG.md を毎 Finding Read）を発見したら、loop に入る前の Step 1 Pre-flight に hoist してキャッシュする。N×M の `jq` / Read 累積を解消すると同時に、resolution の error handling が 1 箇所に集中できる。Simplify 段階の Efficiency 観点で頻出する fix 方向で、producer + consumer 両側が同 pattern を持つ場合は両方とも hoist する
- **disposition enum 拡張ではなく既存 enum の厳格化**: 配布スキルの judgment criteria（reject 一覧 / status enum 等）に新 case を追加する際、新 disposition value を発明する前に **既存値の criteria を厳格化** で表現できないかを検討する。例: triage の "Already addressed in later version" は新 disposition でなく既存 reject 基準 #1「Already addressed」を 2-leg AND test（(i) producer→current 間 CHANGELOG に同 target-skill の fix entry あり、AND (ii) 現 SKILL.md で当該現象が再現しない、両方 cited judgment 必要、片方でも doubt なら fall-through）で具体化する形にした。enum を増やすと downstream parser / mapping table が全部 update 必要になり、後方互換性も崩れる
- **counter 増分は「zero exit only / 失敗 path には without incrementing 明記」**: cleanup eligibility を gate する counter（例: `triage_commit_count`）の increment 命令を SKILL.md に書く場合、commit / commit-error 両分岐に対して symmetry を取る。成功側は `Increment triage_commit_count`、失敗側は `Do not increment triage_commit_count`（または `without incrementing`）を明示する。失敗側で increment 命令を省略するだけだと「省略 = increment しない」と読めるが「省略 = 既存実装由来の暗黙の increment」と誤読される余地が残る。symmetry を明示することで cleanup 条件（`triage_commit_count == 0` で auto-cleanup）の bistability を守る
- **dead flag よりも単一 counter で cleanup 条件を表現**: state 追加で「Step 1 で `*_active = true`、Step 4 で `*_active && *_count == 0` で cleanup」のような boolean flag + counter の合成条件は、counter 単独（`*_count == 0`）で同等の semantics が表現できないかを精査する。今回の triage では fatal abort path（Step 2 の `gh issue list` 失敗等）は Step 4 summary を経由しないため `triage_branch_active` flag が判定に寄与する path が存在せず、counter 単独で OK と判明した。Code Review 段階で「dead flag drop」として上がる頻出 finding パターン
- **Run-level 失敗系は per-Finding edge-case 表に混ぜない**: `references/triage-criteria.md` 等の reference に edge-case 表がある場合、pre-flight 失敗（branch 作成失敗、marketplace.json 不在、detached HEAD 等）を per-Finding edge-case 表に追加すると semantics が混ざる（per-Finding は loop 内、pre-flight は loop 外）。Run-level boundaries 節を分離して書くか、SKILL.md 側の Pre-flight 節で手続き的に記述して reference 側の edge-case 表には載せない。組織化観点として review iter 1 で頻出する findings 方向
- **Step 6 Simplify の rationale 段落は配布スキルでも brevity を保つ**: jq null handling のような subtle behavior の rationale を SKILL.md / references 内で説明する場合、技術的に正確でも段落が膨らみがち（producer 側で 100 words 超）。iter 2 Code Review で「英文 clarity / 冗長」として上がるので、Simplify-revival check（iter k≥2 only）と並んで「rationale prose の brevity」も意識する。配布スキルの prose は技術正確性 + 読みやすさの両立が要件
- **Branch behavior は completion で specify する（negation 禁止）**: SKILL.md の分岐記述（aggregate JSON parse 後の branch ごとの record 書き込み、error class ごとの shared updates、callee-abort vs ok の disposition 差等）で、branch B の動作を「branch A と同じだが X を除く / X 以外」のように **negation で specify** すると hidden gap が発生しやすい（書き手は「除外項目だけ書けば十分」と思うが、後追い読み手は branch A の defaults を必ず正しく解釈する保証がない）。Code Review iter 1 で Critical-severity finding として上がる頻出パターン。(i) 分岐ごとに「**Full <set-name> (applies to both `<status-A>` and `<status-B>`)**」のような shared block を上に立て、(ii) 分岐固有の差分だけを下の「branch-specific actions」で書く 2 階層に整理する。同様に (E) error class（E.1 verdict parse failure / E.2 schema violation / E.3 tool error）の shared updates も「by-completion 10-field explicit list」で書き、negation で済ませない。Class-level extension audit (post-Critical-fix) の対象でもあり、1 箇所 negation を発見したら同 SKILL.md 内の他の分岐記述も全て completion 形に書き直すかを精査する
- **Pre-implementation smoke test (Step 0) for undocumented platform capabilities**: 配布スキル / project-local skill の architectural rewrite で、リポジトリ既存スキルに前例のない platform capability に依存する場合（例: `Skill(...)` を `Agent` dispatch 経由の subagent から呼び出す機能、`subagent_type: general-purpose` のデフォルト tool set 内容、parent allowed-tools の subagent への継承挙動等）、Step 5 Implement に入る前に **Step 0 Pre-implement smoke test** phase を Plan の Test plan に組み込む。実装してから feasibility blocker が判明すると plan 全体が崩れるため。smoke test は (i) feasibility 1-2 項目（実機で動くか PASS / FAIL を返す）、(ii) tool inheritance / availability の確認、(iii) state-management simulation（outer-loop 等の挙動を small fixture で再現）、(iv) audit-trail 記録（READY-TO-IMPLEMENT / NEEDS-FALLBACK / BLOCKED の overall verdict）を含む。Risk 節には「Step 0 通過前提」と明記し、smoke test は「推奨」ではなく「必須 (Step 0 item #N)」と wording する（推奨表記だと iter 2 で Major-severity finding として上がる）。`subagent_type` を明示するか / 暗黙の default に頼るかも smoke test で確認する
- **Dispatch-layer health と callee-layer health の counter 分離**: orchestrator が subagent dispatch 経由で callee skills を呼び出す設計（例: triage の `(D)` per-Finding subagent dispatch + 内部で `Skill(verify-diff)` / `Skill(skill-review)` 等）で、(i) **dispatch-layer error**（subagent 自体の dispatch 失敗 / verdict parse failure / schema violation / tool error — `D_dispatch_error_count` に集約）と (ii) **callee-layer error**（個別 callee skill の error verdict / consecutive-error disable — per-callee `<skill>_disabled` 等）は **counter を分離する**。同じ counter に集約すると Step 4 aggregate で「どちらの layer の health が劣化しているか」が読めなくなる。具体ルール: (E) dispatch-layer error path は `D_dispatch_error_count` のみ advance し、**per-callee disable counter は incremented も reset もしない**（callee 自体は走らなかったため health 情報なし）。consecutive 閾値を超えたら `D_dispatch_disabled = true` で以降の (D) を skip する disable 機構を持つ。callee-layer の `<skill>_disabled` は callee verdict が error / max-iter を返したときのみ操作する。Step 4 aggregate で per-callee disable と (D) disable を **別行で render** すれば dispatch-layer / callee-layer の health observability axis が直交を保つ
- **Sequential N-callee orchestration を 1 Agent dispatch に集約する architectural pattern**: orchestrator が 1 unit of work あたり連続して `Skill(callee-A) → Skill(callee-B) → Skill(callee-C)` のように N 個の callee を順次呼ぶ設計で、callee-side fenced JSON return contract が既に存在しても **orchestrator 側 N decision points の stall surface** が積み上がる構造的問題が観測された場合（reminder / closed list / anti-pattern 列挙系の prose discipline は diminishing returns に入る）、N 個の `Skill()` を 1 個の `Agent` dispatch にまとめる architectural rewrite を検討する。subagent prompt の中で同じ N callee chain を実行し、aggregate JSON 1 個を返す形に再設計すると orchestrator の stall surface が N → 1 に減る（subagent 内 stall surface は残るが per-task scope が極小なので統計的軽減）。設計時の必須事項: (i) aggregate JSON schema は per-callee return field を nest して全 callee 状態を carry する、(ii) (E) dispatch-layer error class（verdict parse failure / schema violation / tool error）を callee-layer error と分離、(iii) `subagent_type: general-purpose` のような明示的指定を Step 0 smoke test で feasibility 確認、(iv) subagent prompt の step 2 wording で「**Do not run further `Skill()` dispatches beyond what is enumerated in this prompt**」のように追加 dispatch を明示禁止して subagent 内 stall surface も bound する
- **Routing-field entry shape validation extension**: 既存ルール（subagent 返却 JSON の per-entry shape validation を parse 時に行う、`Edit` call crash 防止）の **class-level extension**: entry shape の validation 対象は `Edit` の `old_string` だけでなく、**downstream tool call が dereference する全フィールド** に拡張する。例: `publicity_review.remaining_findings[].file` は revert path 計算で `git checkout HEAD -- <entry>.file` に流れるため、(E.2) schema violation 段階で `entries must have non-empty string file` を検証しないと revert 段で `git checkout` が malformed path で crash する経路が残る。canonical pattern は callee return contract に「routing 用フィールド (`file` 等)」が含まれる場合、parse 段階の schema-violation check に entry-shape validation を必ず含めて、downstream で dereference される field は全て non-empty string と検査する。`Edit`-focused のみだと iter 2 Code Review で Major finding として上がる
- **Bold-prose label cross-reference style**: 既存ルール「SKILL.md 内外の相互参照は安定した節見出し・フレーズで記述する」の specialization: 参照先が `### Heading` 形式の見出しではなく **bold-prose label（段落冒頭の `**Per-Finding record kept in memory**` 等）** で identify される場合、`§ <Heading>` だけでなく `§ <Heading>'s "<bold-prose label>" paragraph` のように **bold 内文言を verbatim で囲み込む** form を採る。例: `§ Apply accepted Findings record schema`（存在しない heading）→ `§ Apply accepted Findings's "Per-Finding record kept in memory" paragraph`（実在する bold label）。Code Review で Major-severity な dangling cross-reference として上がりやすい。bold label を後で書き換えると参照側も同期する必要がある（Edit 時の双方向 audit 対象）
- **Sibling enum field の symmetric extension audit**: SKILL.md の record schema で並列に並ぶ enum 値群（例: `record.verify_diff` / `record.skill_review` / `record.publicity_review` がそれぞれ `<status>` 列を enum として持つ）に新値を追加する場合、**全 sibling field に同一の値を追加** することを audit する。`error` のような汎用 disposition value を片方だけに足して他は asymmetric なまま残すと、aggregate render の switch 文 / mapping table で取りこぼしが発生する。Code Review iter 2 / iter 3 の class-level extension audit で「asymmetric enum extension」として finding 化されやすい。逆方向（既存 enum 値の削除 / rename）も同様に sibling field 全体を sweep する
- **Plan rewrite triggered by user material change at Step 4 gate**: dev-workflow Step 4 user-approval gate で user が Decisions の Alternative を選ぶ等の material change を要求した場合、SKILL.md 規定に従って Step 3-(N+1) の追加 review iteration を立ち上げてから ExitPlanMode で再提示する。plan 全体を rewrite した場合（Recommendation/Alternative の swap、Decisions 節の削除、Risks の置換等）は、(i) Title から旧 approach 由来の語句を削除、(ii) Context / Goal / Approach の旧記述を全て新 approach に置換（コスト削減効果が消える等の実態変化も明記）、(iii) Risks 節も新 approach 固有の unknowns に完全置換（旧 approach 由来の Risks は残さない）、(iv) Decisions § N に user 既選択を記録、を 1 pass で実施する。partial rewrite（旧記述を残したまま新記述を併記）は plan size を肥大化させ Step 3-(N+1) reviewer に「buried decisions」「scope creep」と再度指摘される頻出パターン
