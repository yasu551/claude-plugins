# Project Rules - Local (Conversation-Extracted)

このファイルは `--from-conversation` で抽出された project-specific patterns を保持する `.local.md` です。`project.rules.md` の Principles と分離して、organizational rule sharing 時に portable な Principles のみを promote できるよう運用しています。

## Project-specific patterns

- **`jq` の `null` 文字列フォールバック**: 配布スキル内 SKILL.md / references で `marketplace.json` 等から optional フィールドを取り出す際、`jq -r '.x | select(...) | .field' file 2>/dev/null || echo unknown` 形式は **不十分**。`jq` は entry / key 不在時に literal `null\n` を stdout に出して **zero exit** するため、`|| echo unknown` がフォーカルしない。canonical pattern は `value=$(jq -r '(.plugins[] | select(.name == "<n>") | .version) // "unknown"' .claude-plugin/marketplace.json 2>/dev/null)` で **filter 内** の `// "unknown"` を使い、追加で `[ -z "$value" ] && value="unknown"` の post-pipeline `-z` ガードでファイル不在 / `jq` 自体の non-zero exit / 空出力を吸収する 2 段構えにする。producer 側 (issue 本文への version 埋め込み) と consumer 側 (triage の `current_version` 解決) の両方が同一 class なので、片方修正したら必ずもう一方も同 pattern で修正する（class-level extension audit の対象）
- **detached HEAD ガード**: 配布スキルの SKILL.md prose で「現在のブランチを保存してから `git switch -c` で新ブランチに切り替える」フローを指示する場合、`git rev-parse --abbrev-ref HEAD` 単独だと detached HEAD で `HEAD` という literal を返し、後段の `git switch <original_branch>` cleanup が `HEAD` という名前のブランチを探して fail する。Step 1 Pre-flight 等で `git symbolic-ref -q HEAD >/dev/null` の non-zero exit を以って detached HEAD を検出し、early abort で `detached HEAD: cannot save original branch reference` 等の理由を summary に出す。ブランチ操作を伴う routine スキル全般に同じ guard を入れる
- **`refs/heads/<glob>` の single-quote**: SKILL.md prose で `git for-each-ref --sort=-refname refs/heads/triage-* --format='%(refname:short)'` のように glob を含む refspec を書く場合、shell 展開を防ぐため `'refs/heads/triage-*'` で single-quote する。zsh の `nomatch` option がデフォルト有効で、マッチが無いと shell が abort する（routine 走行で silent halting の元）。bash でも quote しておく方が安全。`git for-each-ref` 系の命令を SKILL.md に書く際の必須慣行
- **Run-level invariant の Pre-flight への hoist**: per-Finding / per-iteration loop の中で同一値を毎回 resolve する設計（例: triage の Step 3.3 で marketplace.json から `current_version` を毎 Finding 解決、CHANGELOG.md を毎 Finding Read）を発見したら、loop に入る前の Step 1 Pre-flight に hoist してキャッシュする。N×M の `jq` / Read 累積を解消すると同時に、resolution の error handling が 1 箇所に集中できる。Simplify 段階の Efficiency 観点で頻出する fix 方向で、producer + consumer 両側が同 pattern を持つ場合は両方とも hoist する
- **disposition enum 拡張ではなく既存 enum の厳格化**: 配布スキルの judgment criteria（reject 一覧 / status enum 等）に新 case を追加する際、新 disposition value を発明する前に **既存値の criteria を厳格化** で表現できないかを検討する。例: triage の "Already addressed in later version" は新 disposition でなく既存 reject 基準 #1「Already addressed」を 2-leg AND test（(i) producer→current 間 CHANGELOG に同 target-skill の fix entry あり、AND (ii) 現 SKILL.md で当該現象が再現しない、両方 cited judgment 必要、片方でも doubt なら fall-through）で具体化する形にした。enum を増やすと downstream parser / mapping table が全部 update 必要になり、後方互換性も崩れる
- **counter 増分は「zero exit only / 失敗 path には without incrementing 明記」**: cleanup eligibility を gate する counter（例: `triage_commit_count`）の increment 命令を SKILL.md に書く場合、commit / commit-error 両分岐に対して symmetry を取る。成功側は `Increment triage_commit_count`、失敗側は `Do not increment triage_commit_count`（または `without incrementing`）を明示する。失敗側で increment 命令を省略するだけだと「省略 = increment しない」と読めるが「省略 = 既存実装由来の暗黙の increment」と誤読される余地が残る。symmetry を明示することで cleanup 条件（`triage_commit_count == 0` で auto-cleanup）の bistability を守る
- **dead flag よりも単一 counter で cleanup 条件を表現**: state 追加で「Step 1 で `*_active = true`、Step 4 で `*_active && *_count == 0` で cleanup」のような boolean flag + counter の合成条件は、counter 単独（`*_count == 0`）で同等の semantics が表現できないかを精査する。今回の triage では fatal abort path（Step 2 の `gh issue list` 失敗等）は Step 4 summary を経由しないため `triage_branch_active` flag が判定に寄与する path が存在せず、counter 単独で OK と判明した。Code Review 段階で「dead flag drop」として上がる頻出 finding パターン
- **Run-level 失敗系は per-Finding edge-case 表に混ぜない**: `references/triage-criteria.md` 等の reference に edge-case 表がある場合、pre-flight 失敗（branch 作成失敗、marketplace.json 不在、detached HEAD 等）を per-Finding edge-case 表に追加すると semantics が混ざる（per-Finding は loop 内、pre-flight は loop 外）。Run-level boundaries 節を分離して書くか、SKILL.md 側の Pre-flight 節で手続き的に記述して reference 側の edge-case 表には載せない。組織化観点として review iter 1 で頻出する findings 方向
- **Step 6 Simplify の rationale 段落は配布スキルでも brevity を保つ**: jq null handling のような subtle behavior の rationale を SKILL.md / references 内で説明する場合、技術的に正確でも段落が膨らみがち（producer 側で 100 words 超）。iter 2 Code Review で「英文 clarity / 冗長」として上がるので、Simplify-revival check（iter k≥2 only）と並んで「rationale prose の brevity」も意識する。配布スキルの prose は技術正確性 + 読みやすさの両立が要件
