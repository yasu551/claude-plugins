# Project Rules - Local (Conversation-Extracted)

このファイルは `--from-conversation` で抽出された project-specific patterns を保持する `.local.md` です。`project.rules.md` の Principles と分離して、organizational rule sharing 時に portable な Principles のみを promote できるよう運用しています。

## Project-specific patterns

- **`jq` の `null` 文字列フォールバック**: 配布スキル内 SKILL.md / references で `marketplace.json` 等から optional フィールドを取り出す際、`jq -r '.x | select(...) | .field' file 2>/dev/null || echo unknown` 形式は **不十分**。`jq` は entry / key 不在時に literal `null\n` を stdout に出して **zero exit** するため、`|| echo unknown` がフォーカルしない。canonical pattern は `value=$(jq -r '(.plugins[] | select(.name == "<n>") | .version) // "unknown"' .claude-plugin/marketplace.json 2>/dev/null)` で **filter 内** の `// "unknown"` を使い、追加で `[ -z "$value" ] && value="unknown"` の post-pipeline `-z` ガードでファイル不在 / `jq` 自体の non-zero exit / 空出力を吸収する 2 段構えにする。producer 側 (issue 本文への version 埋め込み) と consumer 側 (triage の `current_version` 解決) の両方が同一 class なので、片方修正したら必ずもう一方も同 pattern で修正する（class-level extension audit の対象）
- **detached HEAD ガード**: 配布スキルの SKILL.md prose で「現在のブランチを保存してから `git switch -c` で新ブランチに切り替える」フローを指示する場合、`git rev-parse --abbrev-ref HEAD` 単独だと detached HEAD で `HEAD` という literal を返し、後段の `git switch <original_branch>` cleanup が `HEAD` という名前のブランチを探して fail する。Step 1 Pre-flight 等で `git symbolic-ref -q HEAD >/dev/null` の non-zero exit を以って detached HEAD を検出し、early abort で `detached HEAD: cannot save original branch reference` 等の理由を summary に出す。ブランチ操作を伴う routine スキル全般に同じ guard を入れる
- **`refs/heads/<glob>` の single-quote**: SKILL.md prose で `git for-each-ref --sort=-refname refs/heads/triage-* --format='%(refname:short)'` のように glob を含む refspec を書く場合、shell 展開を防ぐため `'refs/heads/triage-*'` で single-quote する。zsh の `nomatch` option がデフォルト有効で、マッチが無いと shell が abort する（routine 走行で silent halting の元）。bash でも quote しておく方が安全。`git for-each-ref` 系の命令を SKILL.md に書く際の必須慣行
- **Run-level invariant の Pre-flight への hoist**: per-Finding / per-iteration loop の中で同一値を毎回 resolve する設計（例: triage の Step 3.3 で marketplace.json から `current_version` を毎 Finding 解決、CHANGELOG.md を毎 Finding Read）を発見したら、loop に入る前の Step 1 Pre-flight に hoist してキャッシュする。N×M の `jq` / Read 累積を解消すると同時に、resolution の error handling が 1 箇所に集中できる。Simplify 段階の Efficiency 観点で頻出する fix 方向で、producer + consumer 両側が同 pattern を持つ場合は両方とも hoist する
- **disposition enum 拡張ではなく既存 enum の厳格化**: 配布スキルの judgment criteria（reject 一覧 / status enum 等）に新 case を追加する際、新 disposition value を発明する前に **既存値の criteria を厳格化** で表現できないかを検討する。例: triage の "Already addressed in later version" は新 disposition でなく既存 reject 基準 #1「Already addressed」を 2-leg AND test（(i) producer→current 間 CHANGELOG に同 target-skill の fix entry あり、AND (ii) 現 SKILL.md で当該現象が再現しない、両方 cited judgment 必要、片方でも doubt なら fall-through）で具体化する形にした。enum を増やすと downstream parser / mapping table が全部 update 必要になり、後方互換性も崩れる
- **dead flag よりも単一 counter で cleanup 条件を表現**: state 追加で「Step 1 で `*_active = true`、Step 4 で `*_active && *_count == 0` で cleanup」のような boolean flag + counter の合成条件は、counter 単独（`*_count == 0`）で同等の semantics が表現できないかを精査する。今回の triage では fatal abort path（Step 2 の `gh issue list` 失敗等）は Step 4 summary を経由しないため `triage_branch_active` flag が判定に寄与する path が存在せず、counter 単独で OK と判明した。Code Review 段階で「dead flag drop」として上がる頻出 finding パターン
- **Run-level 失敗系は per-Finding edge-case 表に混ぜない**: `references/triage-criteria.md` 等の reference に edge-case 表がある場合、pre-flight 失敗（branch 作成失敗、marketplace.json 不在、detached HEAD 等）を per-Finding edge-case 表に追加すると semantics が混ざる（per-Finding は loop 内、pre-flight は loop 外）。Run-level boundaries 節を分離して書くか、SKILL.md 側の Pre-flight 節で手続き的に記述して reference 側の edge-case 表には載せない。組織化観点として review iter 1 で頻出する findings 方向
- **Step 6 Simplify の rationale 段落は配布スキルでも brevity を保つ**: jq null handling のような subtle behavior の rationale を SKILL.md / references 内で説明する場合、技術的に正確でも段落が膨らみがち（producer 側で 100 words 超）。iter 2 Code Review で「英文 clarity / 冗長」として上がるので、Simplify-revival check（iter k≥2 only）と並んで「rationale prose の brevity」も意識する。配布スキルの prose は技術正確性 + 読みやすさの両立が要件
- **Branch behavior は completion で specify する（negation 禁止）**: SKILL.md の分岐記述（aggregate JSON parse 後の branch ごとの record 書き込み、error class ごとの shared updates、callee-abort vs ok の disposition 差等）で、branch B の動作を「branch A と同じだが X を除く / X 以外」のように **negation で specify** すると hidden gap が発生しやすい（書き手は「除外項目だけ書けば十分」と思うが、後追い読み手は branch A の defaults を必ず正しく解釈する保証がない）。Code Review iter 1 で Critical-severity finding として上がる頻出パターン。(i) 分岐ごとに「**Full <set-name> (applies to both `<status-A>` and `<status-B>`)**」のような shared block を上に立て、(ii) 分岐固有の差分だけを下の「branch-specific actions」で書く 2 階層に整理する。同様に (E) error class（E.1 verdict parse failure / E.2 schema violation / E.3 tool error）の shared updates も「by-completion 10-field explicit list」で書き、negation で済ませない。Class-level extension audit (post-Critical-fix) の対象でもあり、1 箇所 negation を発見したら同 SKILL.md 内の他の分岐記述も全て completion 形に書き直すかを精査する
- **Pre-implementation smoke test (Step 0) for undocumented platform capabilities**: 配布スキル / project-local skill の architectural rewrite で、リポジトリ既存スキルに前例のない platform capability に依存する場合（例: `Skill(...)` を `Agent` dispatch 経由の subagent から呼び出す機能、`subagent_type: general-purpose` のデフォルト tool set 内容、parent allowed-tools の subagent への継承挙動等）、Step 5 Implement に入る前に **Step 0 Pre-implement smoke test** phase を Plan の Test plan に組み込む。実装してから feasibility blocker が判明すると plan 全体が崩れるため。smoke test は (i) feasibility 1-2 項目（実機で動くか PASS / FAIL を返す）、(ii) tool inheritance / availability の確認、(iii) state-management simulation（outer-loop 等の挙動を small fixture で再現）、(iv) audit-trail 記録（READY-TO-IMPLEMENT / NEEDS-FALLBACK / BLOCKED の overall verdict）を含む。Risk 節には「Step 0 通過前提」と明記し、smoke test は「推奨」ではなく「必須 (Step 0 item #N)」と wording する（推奨表記だと iter 2 で Major-severity finding として上がる）。`subagent_type` を明示するか / 暗黙の default に頼るかも smoke test で確認する
- **Dispatch-layer health と callee-layer health の counter 分離**: orchestrator が subagent dispatch 経由で callee skills を呼び出す設計（例: triage の `(D)` per-Finding subagent dispatch + 内部で `Skill(verify-diff)` / `Skill(skill-review)` 等）で、(i) **dispatch-layer error**（subagent 自体の dispatch 失敗 / verdict parse failure / schema violation / tool error — `D_dispatch_error_count` に集約）と (ii) **callee-layer error**（個別 callee skill の error verdict / consecutive-error disable — per-callee `<skill>_disabled` 等）は **counter を分離する**。同じ counter に集約すると Step 4 aggregate で「どちらの layer の health が劣化しているか」が読めなくなる。具体ルール: (E) dispatch-layer error path は `D_dispatch_error_count` のみ advance し、**per-callee disable counter は incremented も reset もしない**（callee 自体は走らなかったため health 情報なし）。consecutive 閾値を超えたら `D_dispatch_disabled = true` で以降の (D) を skip する disable 機構を持つ。callee-layer の `<skill>_disabled` は callee verdict が error / max-iter を返したときのみ操作する。Step 4 aggregate で per-callee disable と (D) disable を **別行で render** すれば dispatch-layer / callee-layer の health observability axis が直交を保つ
- **Sequential N-callee orchestration を 1 Agent dispatch に集約する architectural pattern**: orchestrator が 1 unit of work あたり連続して `Skill(callee-A) → Skill(callee-B) → Skill(callee-C)` のように N 個の callee を順次呼ぶ設計で、callee-side fenced JSON return contract が既に存在しても **orchestrator 側 N decision points の stall surface** が積み上がる構造的問題が観測された場合（reminder / closed list / anti-pattern 列挙系の prose discipline は diminishing returns に入る）、N 個の `Skill()` を 1 個の `Agent` dispatch にまとめる architectural rewrite を検討する。subagent prompt の中で同じ N callee chain を実行し、aggregate JSON 1 個を返す形に再設計すると orchestrator の stall surface が N → 1 に減る（subagent 内 stall surface は残るが per-task scope が極小なので統計的軽減）。設計時の必須事項: (i) aggregate JSON schema は per-callee return field を nest して全 callee 状態を carry する、(ii) (E) dispatch-layer error class（verdict parse failure / schema violation / tool error）を callee-layer error と分離、(iii) `subagent_type: general-purpose` のような明示的指定を Step 0 smoke test で feasibility 確認、(iv) subagent prompt の step 2 wording で「**Do not run further `Skill()` dispatches beyond what is enumerated in this prompt**」のように追加 dispatch を明示禁止して subagent 内 stall surface も bound する
- **Routing-field entry shape validation extension**: 既存ルール（subagent 返却 JSON の per-entry shape validation を parse 時に行う、`Edit` call crash 防止）の **class-level extension**: entry shape の validation 対象は `Edit` の `old_string` だけでなく、**downstream tool call が dereference する全フィールド** に拡張する。例: `publicity_review.remaining_findings[].file` は revert path 計算で `git checkout HEAD -- <entry>.file` に流れるため、(E.2) schema violation 段階で `entries must have non-empty string file` を検証しないと revert 段で `git checkout` が malformed path で crash する経路が残る。canonical pattern は callee return contract に「routing 用フィールド (`file` 等)」が含まれる場合、parse 段階の schema-violation check に entry-shape validation を必ず含めて、downstream で dereference される field は全て non-empty string と検査する。`Edit`-focused のみだと iter 2 Code Review で Major finding として上がる
- **Bold-prose label cross-reference style**: 既存ルール「SKILL.md 内外の相互参照は安定した節見出し・フレーズで記述する」の specialization: 参照先が `### Heading` 形式の見出しではなく **bold-prose label（段落冒頭の `**Per-Finding record kept in memory**` 等）** で identify される場合、`§ <Heading>` だけでなく `§ <Heading>'s "<bold-prose label>" paragraph` のように **bold 内文言を verbatim で囲み込む** form を採る。例: `§ Apply accepted Findings record schema`（存在しない heading）→ `§ Apply accepted Findings's "Per-Finding record kept in memory" paragraph`（実在する bold label）。Code Review で Major-severity な dangling cross-reference として上がりやすい。bold label を後で書き換えると参照側も同期する必要がある（Edit 時の双方向 audit 対象）
- **Symmetric / cross-file extension audit (sibling enum + SKILL.md ↔ `references/*.md`)**: 同 SKILL.md 内で並列に並ぶ enum 値群（例: `record.verify_diff` / `record.skill_review` / `record.publicity_review` が `<status>` 列を enum として持つ）に新値を追加する場合、**全 sibling field に同一の値を追加** することを audit する（`error` のような汎用 disposition value を片方だけに足して他は asymmetric なまま残すと、aggregate render の switch 文 / mapping table で取りこぼし）。逆方向（既存 enum 値の削除 / rename）も同様に sweep。**Cross-file 兄弟**: SKILL.md に **parse-error condition の closed list** / disposition enum / status set 等を持ち、同期した同列挙が `references/<skill>-criteria.md` 等の reference table（Edge-case 表、Disposition mapping 表等）に二重管理されている場合、新 case 追加 / 既存 case 修正は **必ず両ファイルを同期更新**: (i) **SKILL.md 側 closed list の Edit 直後に reference file の table 行を Read → 同 case 行を追加 / 修正する 2-step audit**、(ii) review 段階で見落とした **pre-existing instance**（リストには既存だが reference table 側だけ古い）も sweep 対象、(iii) source of truth が SKILL.md 側にあるなら reference 側に `Source of truth: SKILL.md § ...; this table mirrors that list — keep in sync` のような sync directive 行を 1 行入れて bidirectional 追従義務を明文化。実例: triage で SKILL.md parse-error 条件に新 case「Zero `### Finding` headings」を追加したが `references/triage-criteria.md` Edge-case 表に同 case 追加し忘れ Code Review iter 2 で Major finding。Code Review iter 2 / iter 3 の class-level extension audit (post-Critical/Major-fix) の対象であり、1 case 漏れを発見したら同 SKILL.md 内の他の closed list / sibling reference table 全てを sweep する
- **Plan rewrite triggered by user material change at Step 4 gate (+ Recommendation/Alternative swap specialization)**: dev-workflow Step 4 user-approval gate で user が Decisions の Alternative を選ぶ等の material change を要求した場合、SKILL.md 規定に従って Step 3-(N+1) の追加 review iteration を立ち上げてから ExitPlanMode で再提示する。plan 全体を rewrite する手順は (i) Title から旧 approach 由来の語句を削除、(ii) Context / Goal / Approach の旧記述を全て新 approach に置換（コスト削減効果が消える等の実態変化も明記）、(iii) Risks 節も新 approach 固有の unknowns に完全置換（旧 approach 由来の Risks は残さない）、(iv) Decisions § N に user 既選択を記録、を 1 pass で実施。**Swap-direction specialization** (user が Alternative を選んだケース): (a) **Recommendation と Alternative の position を swap** して新 Recommendation が user 選択を反映、(b) **「user 既選択」または同等の annotation を 新 Recommendation の見出しに追加**（後追い triage / Step 3-(N+1) reviewer が「user が明示的に選んだ option」と読み取れる）、(c) **swap に伴って Risks / CHANGELOG note も新 Recommendation 文脈に書き換え**（例: 「default false の silent behavior change avoidance」が「default true の全 user behavior change 波及」に置換）、(d) **Approach / Scope / Test plan 節も同期更新**（例: scope file 数調整）。partial rewrite（旧記述を残したまま新記述を併記）は plan size を肥大化させ Step 3-(N+1) reviewer に「buried decisions」「scope creep」と再度指摘される頻出パターン
- **Routine-side action ownership over environment-finalization delegation**: 非対話／ルーチン実行を想定するスキル（`dev-workflow-triage` 等、Claude Code on the Web の routine 想定）が外部副作用 (`git push` / artifact 公開 / 通知発火等) を伴う場合、SKILL.md 内で **「session finalization 等の環境機能に委譲する」と宣言しない**。`triage` 旧設計の `git push is not performed here — Claude Code on the Web's session finalization handles pushes.` 宣言は、(i) session-level user instructions が別ブランチ名を "designated branch" として push 指示すると orchestrator が consolidate + 別名 push に倒れる誤読の余地を残し、(ii) local 環境（session finalization が無い）では暗黙に push が落ちるという feature parity 欠落を生んだ。canonical pattern: routine 自身が外部副作用を **once per run** で実行し（位置は Step 4 末尾等の単一箇所、cleanup 判定後 / summary stdout 直前）、`allowed-tools` に必要 glob（`Bash(git push *)` 等）を追加、session-level の "designated branch" rule との conflict 文脈を SKILL.md 内に明記して `consolidate into a different name loses same-day re-run stacking semantics and disconnects the operator's PR identity from the run timestamp` 等の rationale を伴う禁止条項を置く。「環境が代行する」前提は配布性 / 環境差異耐性の両面で破綻するため、routine が own すべき副作用は routine 内で完結させる
- **Closed form-set invariant for multi-form summary tokens**: SKILL.md の summary line（例: `triage-branch: <name> (based on <base>) — <details>`）に新 outcome を追加する場合、`triage-branch:` の render form 数を **closed list として明示的に bounded** に保つ。partial-failure / cleanup 中断のような「主 outcome は確定したが二次 cleanup が一部失敗」の状態を新 form として加えると form 数が爆発し、後追い読み手が switch 漏れする。canonical pattern は (i) 主 outcome（success / fail / 0-commit auto-cleanup）の form 数を 3 以下程度に bounded し、(ii) 二次 cleanup の partial 失敗は別 warning line（`cleanup: switch back failed (...)` / `cleanup: branch -D failed (...)`）に分離する。SKILL.md prose で「the `triage-branch:` form set stays closed at 3 — partial-cleanup failures surface via separate `cleanup:` warning lines」のように form-set 数を明示文書化すると class-level extension audit の対象が明確になる
- **External-command failure: retry once + stderr-derived `<reason>` + no auto-recovery**: SKILL.md 内で外部 CLI（`git push` / `gh api` / `curl` 等）の失敗を扱う非対話 routine では、retry / `<reason>` 抽出 / リカバリ姿勢の 3 点を canonical pattern で書く。(i) **retry は 1 回のみ、1–2 second sleep**（指数バックオフ 4 段は over-spec — `git push` 失敗の主因は auth / non-fast-forward / pre-push hook rejection 等 deterministic で、retry で改善しない。transient network error のみ retry が効き、それは 1 回で十分）。(ii) **`<reason>` は stderr の最終 non-empty 行 を ≤ 80 文字に truncate**、stderr が空または whitespace-only なら `(no stderr)` で fallback（ロケール依存の文言 parse は避け、operator が auth / network / non-fast-forward / hook を区別できる程度の signal を rendering）。(iii) **auto-recovery は明示的に禁止**（`Do not auto-recover (no force push, no rebase, no branch rename) — the operator can <command> manually post-run`）— 禁止文言が無いと downstream LLM が「rebase してから retry」等を hallucinate する。retry 仕様の rationale 段落（"retry primarily helps transient network errors; deterministic rejections will not recover"）は Step 6 Simplify で narration として削られるので prose に残さない（Simplify-revival check の対象）
- **Mis-justification audit when citing infrastructure mechanisms**: 設計上の制約 / 禁止条項を別 mechanism の挙動を引用して justify する場合（例: 「`§ Triage branch isolation` の `git for-each-ref` requires prior-run `triage-*` to survive on origin」）、引用先 mechanism の実挙動を **必ず一次情報** で確認する。実例: triage の旧 prose は `for-each-ref` が origin を読むと暗示していたが、実際は local refs のみ参照。誤引用が peer review iter 1 で Major finding として上がった。canonical fix: (a) cited mechanism の実挙動を verify、(b) 誤りなら justification を design intent ベース（例: `consolidating into a different name loses same-day re-run stacking semantics and disconnects the operator's PR identity from the run timestamp`）に書き換える。一次情報を引かずに mechanism reference で justify する prose は、後追い検証コスト＋誤導リスクの両方が高い
- **Cross-section rendering-authority pointer (computed-here / rendered-there)**: SKILL.md 内で「state value を section A で compute し、summary rendering を section B で行う」設計を採る場合、両 section に **明示的な相互参照ポインタ** を置く。実例: `§ Auto-cleanup` で 0-commit form を render するか判定し、`§ Push triage branch to origin` で `pushed to origin` / `push-failed (<reason>)` suffix を決める設計では、(i) `§ Auto-cleanup` 冒頭に `Run before § Push triage branch to origin and the summary stdout (Auto-cleanup determines the rendering of the 0-commit form only — the >0-commit form's suffix is decided by § Push triage branch to origin below)`、(ii) `§ Push triage branch to origin` 冒頭に `Run after the auto-cleanup decision settles, before emitting the summary stdout (so the triage-branch summary line above can render the post-push state)` のような完成形ポインタを置く。実行順 + 各 section の rendering 担当範囲が両側から読めるので、後追い読み手が「どの section が最終 render を確定するか」を曖昧にしない
- **Agent definition (`.claude/agents/<name>.md`) frontmatter `allowed-tools` requirement**: 既存ルール「Pattern A iteration loop skill の `allowed-tools` baseline mirror」の **class-level extension**: 配布される SKILL.md だけでなく、project-local agent 定義ファイル（`.claude/agents/<name>.md`、Claude Code の subagent として `Agent` ツールから `subagent_type: <name>` で dispatch されるもの）も同じ落とし穴を持つ。frontmatter に `allowed-tools` 宣言が無いと、subagent が起動しても内部で `Skill()` / `Agent` / `Bash(...)` のような nested dispatch が permission denied になる（フロントマター上 `name` / `description` のみで通ってしまうため、SKILL.md と違ってテスト系で検出されにくく、実 routine 走行で初めて failure が surface する）。canonical pattern は (i) agent が dispatch する callee skills（`Skill(verify-diff)` / `Skill(skill-review)` / `Skill(publicity-review)` 等）の `Skill(<name>)` 参照を明示、(ii) callee skills の `allowed-tools` の **union** を取り、`Read` / `Edit` / `Agent` / `TodoWrite` / `Bash(...)` のような nested 実行に必要な tool を全て列挙、(iii) `Bash(*)` を避け callee と同じ glob 粒度（`Bash(git diff *)` / `Bash(git rev-parse *)` / `Bash(git checkout HEAD -- *)` 等）に合わせる、の 3 点を満たす。新規 agent 追加時 / callee 構成変更時は、(a) 該当 agent が dispatch する全 callee skills の SKILL.md frontmatter `allowed-tools` を 1 行ずつ diff、(b) union を agent frontmatter に書き写す、の 2-step audit を踏む（callee 1 つの `allowed-tools` 拡張に追従し忘れると nested dispatch が silent fail する）
- **Subagent inline-execution prohibition for `Skill(<callee>)` dispatches in agent definitions**: agent 定義 (`.claude/agents/<name>.md`) で `allowed-tools` に `Skill(<callee>)` を含める場合、subagent dispatch 時に **callee の SKILL.md 内容が subagent の context に可視化される** という副作用がある（Claude Code が `Skill()` を allowed にすると callee SKILL.md を inject するため）。これにより subagent は `Skill(<callee>)` tool call を **dispatch** するのではなく、callee の logic を **inline で再実行** して callee 内部の verdict JSON を直接返してしまう failure mode が発生する。実例: `triage-per-finding-reviewer` agent が `Skill(verify-diff)` を tool dispatch せずに verify-diff の Workflow を inline 実行し、aggregate JSON `{"status": "ok|callee-abort|error", ...}` ではなく verify-diff 内部の `{"status": "converged", "iterations_used": 1, ...}` を返した結果、orchestrator (E.2) schema violation で 3 連続 fail → `D_dispatch_disabled = true` に到達して残り全 Finding が dispatch skip された。canonical mitigation: agent 定義の `## Inputs` と `## Flow` の間に `## Dispatch discipline` 節を新設し、(i) **`MUST` 級の明示宣言** — 「Each callee MUST be invoked via its `Skill(<name>)` tool call. Do not read, interpret, or replicate any callee's SKILL.md logic inline — even if the callee's SKILL.md content is visible in your context. Do not construct or simulate callee verdicts」、(ii) **concrete restatement** — 「when the Flow says "dispatch `Skill(verify-diff)`", issue a `Skill(verify-diff)` tool call and wait for its return. Do not substitute your own evaluation of the diff, scenario generation, or verdict construction」、(iii) **closed-list bound** — 「Do not run further `Skill()` dispatches beyond the N enumerated above」（既存ルール「Sequential N-callee orchestration」の (iv) と同 wording で structural 整合）の 3 段落で reinforcement-by-repetition を行う。3 段落の意図的重複は `§ No-Stall Principle` return-point reminder と同じ pattern（抽象節だけでは agent が決定の瞬間に参照しないため、決定点の bullet 直後に置くことで違反確率を下げる）で、redundancy ではなく positive obligation / concrete example / closed-set bound の 3 angle 補強。anti-pattern として callee 内部スキーマの具体的 field 名を agent prose に追加で書き出すことは **禁止**（inline 実行の材料を増やすリスクがある）。新規 agent 追加 / `allowed-tools` に `Skill(<name>)` を追加する変更では `## Dispatch discipline` 節の有無を必ず audit する
- **Routing-identifier permissive / content strict layering**: SKILL.md / references で「issue / 入力データを triage / routing 用に identify する fields（title format、識別子 regex、grammar 揺れに晒される表層 token 等）」と「triage / judgment ロジックが直接参照する内容 fields（Finding 内 enum、required structural fields、判定に使う数値 / カテゴリ等）」が併存する場合、**identifier layer は permissive、content layer は strict** に分離して設計する。同 SKILL.md 内で両 layer に同水準の strict 検証を当てると、identifier 側の grammar 揺れ（例: `1 finding` vs `N findings` の単複、`Findings: N` trailer の有無）で本来 triage 可能な input が silent skip / 半 silent skip される regression 経路が開く。canonical 判断軸: (i) **対象 input が事前にフィルタ済み chamber**（例: triage 専用 repo / 専用 channel）なら identifier 側の format check は **不要**、body / content parse を canonical discriminator に倒す。(ii) identifier を保持する必要がある場合は permissive form（`\d+ finding(s)?` のような単複両対応、trailer は optional cross-check）にし、不在 / 揺れだけでは reject にしない。(iii) content layer（Finding section 内の 4 fields、Target skill / Category enum 等）は judgment に直接利用されるため strict を維持し、緩めない。design rule of thumb は「identifier は permissive、content は strict」を SKILL.md prose に明文化して、後追い読み手 / reviewer が両 layer を同じ厳格度で設計する誤りを防ぐ。format check 廃止が大きな scope 変更になる場合は、Step 4 user gate で「identifier 廃止 vs 緩める」の 2 案を Decisions に並べて user 判断を仰ぐ
- **Vacuous-truth gap in per-element ALL-quantifier close predicates**: SKILL.md で「ある決定 (auto-close / cleanup / acceptance 等) を per-element ALL-quantifier predicate で表現する」場合（例: `every Finding is OK to close` / `all records pass check` / `every entry succeeded`）、**zero records 時に predicate が vacuously true** になり、本来 close すべきでない whole-state（whole-issue parse-error / empty input / dispatch-aborted 等）が silently 同じ branch に倒れる経路が開く。実例: triage の close decision を `every Finding is accepted (or rejected with reason cited)` で表現したとき、parse-error で Finding 配列が空のまま close 判定に入ると vacuous all-pass で auto-close が起動した。canonical fix: close / cleanup 系 decision は **2-leg AND predicate** で書く — (i) **whole-state gate**（"the issue is in a closable whole-state — not parse-error, not dispatch-aborted, etc."）、AND (ii) **per-element check**（"every Finding satisfies <X>"）。leg (i) で zero-record / abort-state を明示的に弾いてから leg (ii) を評価する。SKILL.md prose には「per-element ALL-quantifier alone is insufficient — must be gated by whole-state predicate to avoid vacuous-true regression on empty / aborted state」と vacuous-truth 注記を 1 行入れて、後追い読み手が同 pattern を増やしたときの落とし穴を塞ぐ。Code Review iter 1 で Critical-severity finding として上がる頻出パターン（Step 3 plan review は behavior を抽象的に扱うため、predicate logic の vacuous gap が plan 段で漏れて Step 8 で初めて catch される）
- **Step renumbering propagation in SKILL.md (mechanical sweep + individual overrides)**: SKILL.md に **新しい Step を挿入** または **既存 Step を再番号化** する変更は、Step 番号が SKILL.md 本文 / `references/*.md` / `README.md` / `CHANGELOG.md` cross-ref 等で多数参照されているため、plan の Design 節で **2 階層に分けて記述** する必要がある: (i) **Generic rule (mechanical sweep)** — 旧 Step N → 新 Step M の単純置換が安全な参照群を列挙する（例: `旧 Step 9 (Update Rules) → 新 Step 11` / `旧 Step 9.5 → 新 Step 11.5` / `旧 Step 10 (Completion Hooks) → 新 Step 9`）。(ii) **Individual overrides** — generic sweep を適用すると意味的誤りになる箇所を line 別に列挙する。代表的な誤り class: 旧 `Runs after Step N` が新番号で当該 hook 自体が Step N になるケース（→ `Runs as Step N (immediately after Step N-1)` 形に明示書き換え）、Step 8 の return-point reminder で `Step 9 transition` のような曖昧化が起きるケース（→ `Step 9 (Completion Hooks) transition` で disambiguate）、`runs after Step 10` が `Step 10` の semantic 移動とともに意味が変わるケース。**line 番号はすべて pre-edit reference** と明記して、edit 進行に伴う行番号 drift を防ぎ、最終確認は phrase / heading anchor で行う旨を Test plan に書く。Step renumbering が `references/*.md` に波及する場合は § J 形式で line-by-line checklist を作る
- **TodoWrite single-row for unknown sub-iteration count at registration time**: Step 1 sub-step 7 で TodoWrite に phases を登録する設計で、**sub-iteration count が user-approval input に依存する** Step（例: 新 Step 10 Interactive Commits の commit 数は commit-plan approval gate を通過して初めて確定する）は、**Step N-1 ... Step N-M の per-iteration row に展開せず single row のまま登録** する。理由: (i) 登録時点で M が未確定なので展開できない、(ii) 後から展開すると TodoWrite 行を mid-flight で書き換える必要があり stall surface を増やす、(iii) per-commit / per-iteration の進捗は Step 内 prose で `Per-commit loop` 等の inline 構造で扱う方が semantic 整合する。SKILL.md Step 1 sub-step 7 の phase enumeration に「`Step <N>: <Name>` (single row — per-X iteration is handled inline within Step N because the X count is not known until <reference paragraph>)」のような annotation を 1 行入れて、後追い読み手 / reviewer が「なぜ展開していないか」を読み取れるようにする。既存 Step 3 / Step 8 review iteration (count = N from `review_iterations` config) は count が registration 時に確定しているため per-iteration 展開する — 両 pattern を混同しないこと
- **Behavior-change default-flip → CHANGELOG opt-out note + downstream automation visibility**: 配布されるスキルの config flag の **default value を flip** する変更（例: `interactive_commits` を default `false` → `true` に変える）は、minor semver bump 単独では signal 強度が不足する場合がある。`hooks.on_complete` 直後の挙動が変わる等、downstream automation（user 側で「dev-workflow 終了時の uncommitted state」依存の wrapper / CI 等）に波及する可能性があるため、CHANGELOG entry には **2 つの要素** を明示する: (i) **opt-out 手順を冒頭で loud に表記** — `**Default: enabled** — set \`<flag>: <old-default>\` in \`<config-path>\` to opt out` の形式で、entry の最初の 1 文に置く。"merge note にあとで書いておく" は読まれない。(ii) **downstream automation note** — Risks 節 / CHANGELOG note で「`<新挙動>` 依存の user 側 automation がある場合 minor bump では signal 強度不足の可能性 → CHANGELOG note で visibility 確保」のような明示的注意喚起を入れる。minor bump 自体は compatibility signal として機能するが、自動 update する CI 等は CHANGELOG を読まずに走るため、signal 強度を補完する prose が必要。Step 4 user gate で default を flip する判断をした場合は、この 2 要素を Decisions 該当項目の rationale に直接組み込む
- **Multi-form gate vocabulary disambiguation in § No-Stall Principle enumeration**: 新 Step が **複数の user-facing gate** を持ち、各 gate の response taxonomy が異なる場合（例: Step 10 Interactive Commits の (a) commit-plan approval gate / (b) per-commit accept gate (accept/adjust/cancel/NOT-approval) / (c) fold-or-defer gate (fold/defer binary) / (d) ambiguous-adjust clarifier）、§ No-Stall Principle の user-gate 列挙を **gate ごとに別 bullet** で列挙する（merge して 1 行にしない）。各 bullet で、当該 gate の canonical token classifier がどの段落で defined されているかを `§ Step <N>'s "<paragraph label>" paragraph` 形式で明示参照する（gate vocabulary が SKILL.md 内 1 箇所で defined されることを保証）。conflate すると後追い読み手が「どの gate でどの enum を使うか」を取り違える。Code Review iter 3 で Major-severity gate count 不整合として上がる頻出パターン（per-commit accept / fold-or-defer / ambiguous-adjust を per-commit gate に統合してしまうと、fold/defer の binary semantics が accept/adjust/cancel の 4 値 enum に巻き込まれて wrong action 経路が開く）
- **Filename-robust git collection + per-commit pathspec scoping for Step 10 / commit flows (canonical form + untracked Read fallback + stop-hook safe commit)**: SKILL.md prose で `git status` を parse して downstream `git add` / `Read` / `git diff` / `git commit` に渡す flow を記述する場合、canonical form は `git status --porcelain=v1 --untracked-files=all -z`: (i) `--porcelain=v1` は format を pin（v2 への upgrade で stability を確保）、(ii) `--untracked-files=all` は user 環境の `status.showUntrackedFiles=no` を override、(iii) `-z` は NUL-separated entry を出力し C-style quoting を抑制（space / quote / non-ASCII 含む filename も verbatim 復元可能）。`-z` を省くと downstream `git add -- "<filename>"` で C-quoted 文字列が pathspec 不一致を引き起こす。**Untracked Read fallback for "all-changes diff"**: `git diff <base-commit>` は untracked files を含まないため、(i) tracked files は `git diff <base-commit>` の per-file portion、(ii) untracked files は `Read` で全内容を取得し "new file" hunk として present する 2 経路を明示。`??` prefix を untracked discriminator に使い、Procedure 冒頭で **once** 実行して in-memory roster + per-file diff を保持 → per-commit loop で reuse。UI prose で「added file」と「modified file」を区別表示する。**Multi-pathspec single call + `--` separator + double-quote convention**: `git add -- "<file-1>" "<file-2>" ...` を single invocation で発行（一括）、`-A` / `git add .` は unrelated drift を staging するため禁止。**Stop-hook auto-stages unrelated files: scope each commit with `git commit -m <msg> -- <paths>`**: Web 環境の `~/.claude/stop-hook-git-check.sh` 等が turn 境界で untracked files（`.claude/plans/*.md` 等の作業 staging 文書）を `A` prefix で auto-stage する場合、`git commit -m <msg>` 単独だと unrelated staged files が混入する。canonical mitigation: (i) **`git commit -m <msg> -- <path-1> <path-2> ...`** で各 commit を pathspec で scope（flag order に注意: `-m` は `--` より前に置く、逆だと ambiguous で fail）、(ii) 作業 staging 文書を含む directory は `.gitignore` に追加せず scope-out として明示除外（gitignore に入れると team-shared な plan が消える）、(iii) commit-plan proposal の **除外（untracked だが本 PR scope 外）** 節で明示
- **Counter lifecycle + counter-based routing (zero-exit-only increment + amend exclusion + actuality routing)**: SKILL.md prose で「成功した landing 単位を数える counter」（例: Interactive Commits の `landed_count`、triage の `triage_commit_count`、dispatch の `dispatch_count`）を持つ場合、canonical lifecycle を **4 点 explicit** で記述する: (i) **loop 開始時に 0 で initialize**、(ii) **landing operation の zero exit 時のみ +1**、(iii) **retry / abort / commit-failed path では incremented しない**、(iv) **`--amend` / re-commit で同じ logical unit を更新した場合は再 increment しない**（unit は initial zero-exit landing で既に数えられているため）。SKILL.md prose に「On zero exit, **increment `<counter>` by 1**」「The amend re-commits the same logical commit — `<counter>` is **not** re-incremented」のような symmetric / 明示文言を入れる。**Downstream routing は config-flag ではなく counter を判定軸にする**: Completion / cleanup / next-action 系の routing で「PR 開いてから resume / commit してから resume」のような分岐を書く場合、**実際の counter（`landed_count > 0`）を判定軸にする** — config-flag（`interactive_commits: true` 等）の **intent** 値で routing すると、`interactive_commits: true` だが Step 10 skip 条件（`git diff --quiet` + untracked empty）で 0 commits 終了した case を取り違える。intent flag は「やる予定だった」、counter は「実際に何が起きたか」。SKILL.md prose で「`landed_count > 0`: <Step で commit 済み> ... `landed_count == 0` (either because `<flag>: false` or because <Step> was skipped): ...」のように **counter の値 → 2 つの異なる cause を OR で union** して書く。判定軸選択を間違うと「skipped-but-enabled」case が silent regression する
- **2-stage grep audit for cross-file Step renumbering**: SKILL.md / references / README / CHANGELOG にまたがる Step 番号 renumbering の最終確認は **2 段階 grep** を canonical procedure とする: (1) **phrase-pair-only grep**（`Step 9: Update Rules|Step 9\.5: Self-Retrospective|Step 10: Completion Hooks` のような旧 section heading 句の OR）で 0 hit を確認 — full-phrase の取り残しを検出、(2) **word-boundary grep on bare numbers**（`\b(Step 9|Step 9\.5|Step 10)\b`）で残り hit を目視確認 — 新番号 semantics 下で正規に「Step 9 (Completion Hooks)」「Step 10 (Interactive Commits)」を指す参照は OK、誤って旧番号のまま残った参照は flag。phrase-only grep だけだと bare number 参照が漏れる、bare-number grep だけだと正規参照も全部 hit して noise が多い。2 段階構成で false-negative / false-positive を両方抑える。Test plan 5 や Design § J で line-by-line checklist と組み合わせて記述し、**line 番号はすべて pre-edit reference** と明記する（edit 進行に伴う行番号 drift を防ぐ意図表明）
- **Token defined-once + cross-reference (re-render 禁止)**: SKILL.md で **localized token / status token / enum mapping** のような「複数箇所から参照される token」を扱う場合、**1 箇所に定義 + 他箇所は cross-reference** の形を採る。具体例: `§ Step 10` で `Localized summary tokens` を 1 箇所 defined すれば、`§ Completion` の partial-state line render は `emit the localized partial-completion token defined at § Step 10's "Localized summary tokens" paragraph` のように **re-render せず参照** する。re-render すると canonical 側の更新が他箇所に伝播せず drift する（i18n token は特に高頻度で wording 更新が入るため、drift リスクが高い）。cross-reference 構文は既存ルール `Bold-prose label cross-reference style` の `§ <Heading>'s "<bold label>" paragraph` 形式を使う。Simplify 段階で「Completion 節に同じ token を再記述している」finding が出るので、Step 6 で発見次第 1 source-of-truth に統合する

- **Stall mitigation: callee-side wording (A) + orchestrator-side pre/return-point reminders (B) — 2-stage serial pair**: `Skill(<callee>)` dispatch 後 stall は **(A) callee 側 wording fix** と **(B) orchestrator 側 reminder** をセットで適用する必要があり、片方だけだと partial fix で終わる（(A) が成立していない状態で (B) だけ入れても callee の prompt-injection が override 効果を相殺）。**(A) Callee-side schema-verb wording**: callee 側 SKILL.md で the response, matching the schema for the mode that ran:` のように **`Emit ... at the end of the response`** 形式で JSON の schema 位置を述語化（`End every invocation` / `Terminate` / `Stop` 等の action 動詞は禁止）、(ii) and emitting it does not terminate the orchestrator's turn」** + 「The same agent that ran this skill must immediately issue the next tool call dictated by the orchestrator's flow」+ 「Do not insert a prose summary, an acknowledgment, or a "shall I proceed?" sentence between the JSON verdict and the next tool call」、(iii) **uniqueness clause** —「Only one fenced JSON block — the verdict block — appears in the response, so callers can locate it unambiguously.」。同 directive を 3 callee に置く場合は **byte-identical wording**（`Skill(<this-skill>)` token のみ差し替え）で配置。新規追加 / 編集時は terminal-action verb の grep を必ず行う。**(B) Orchestrator-side entry + return-point reminder (orthogonal coverage)**: return-point reminder（callee return 後の bullet 直後）だけでは「決定の瞬間」をカバーできない経路がある — callee 側 prompt-injection が orchestrator の decision boundary を turn-end に倒すケースで return-point reminder は **読まれた時点で既に turn が閉じている**。canonical mitigation は **entry-side `**Pre-invocation reminder**`** を `Skill(<callee>)` dispatch **直前**に挿入し return-point reminder と直交させる。reminder 構造: (i) **next tool call の named specification** —「the next tool call after `Skill(<callee>)` returns is `Skill(<next-callee>)` (on `<status-A>` / `<status-B>`) or the (e) error-handling branch (on `error` / verdict missing-or-malformed)」、(ii) **JSON verdict framing as return value** —「Treat <callee>'s JSON verdict as a return value to parse, not a turn boundary — emit the parse → status branch → next dispatch as the next tool call」、(iii) **stable cross-reference** — `See § No-Stall Principle.`。entry / return-point の duplication は **意図的な reinforcement-by-repetition**（`Subagent inline-execution prohibition` の 3-paragraph repetition と同思想 — 抽象節だけでは agent が決定の瞬間に参照しない）で Simplify-revival check で削減候補に上げない旨を SKILL.md prose に 1 行 rationale - **Anchor-mismatch sibling pre-extraction for sibling-symmetric directive placement**: 既存 inline directive（段落 / bullet 内に埋まった prose）を **独立 top-level `## <Section>` に切り出して 3 callee に sibling-symmetric placement** する設計で、各 callee の anchor section 構造が **非対称** な場合（例: 2 callee は既に top-level `## Stop hook structural conflict (caller-side note)` section を持つが、1 callee は `## Scope` 配下の bullet にだけ書かれている）、**uniform placement 不可能** な anchor-mismatch が発生する。canonical fix: (i) anchor が bullet レベルでしか存在しない sibling skill を **先に top-level section に promote** する pre-extraction step を Design に組み込む（bullet wording は保持しつつ、配置を昇格）。(ii) その後で 3 callee 全てに新独立 section を「`## <existing-top-level-section>` の **直前**」のような uniform position に挿入する。(iii) bullet 切り出し時には元 bullet を「`## Scope` 配下に retain」と「no-stall 部分は新 `§ Sub-skill caller directive` への 1 行 cross-reference に圧縮」の 2 経路に split する（scope info と no-stall directive を semantic に分離）。Step 3 plan review iter 2 で anchor mismatch を blocker C-severity finding として発見しやすく、pre-extraction を Design に書かないまま implementation に入ると Edit 時に「どこに置けば 3 callee 横断で揃うか」が判定不能になる。class-level extension audit の対象: sibling-symmetric extension を計画する時は **全 sibling の anchor section 構造を pre-implementation で 1 行ずつ diff** し、structural asymmetry があれば pre-extraction を Design 節に explicit step として組み込む
- **Routine-skill anxiety-abort prohibition + canonical scale-management gate enumeration**: 非対話／ルーチン実行を想定するスキル（`dev-workflow-triage` 等、複数 callee を multi-iter で dispatch する設計）で **mid-flow abort 誘惑** が観測された場合、SKILL.md `§ No-Stall Principle` 末尾に **(i) workload-anxiety abort is forbidden + (ii) 当該 skill が持つ scale-management gate を closed list で enumeration** する形の明示禁止条項を入れる。canonical pattern: (a) **禁止行為の wording** — 「Aborting the routine mid-flow because cumulative `Skill()` loads "feel expensive", because the per-Finding sub-flow "looks long", or because of any equivalent operational anxiety is a No-Stall Principle violation」、(b) **当該 skill の scale gate を citation 付き列挙**（例: triage では `--limit 50` cap / per-callee `*_disabled` flag / `(d-loop)` 早期退出 + max-iter cap の 3 種）、(c) **anti-pattern self-recognition prompt** — 「If you find yourself reasoning "X Findings × Y callees would consume too much context — let me stop here", that is precisely the anti-pattern this clause forbids」。abort triggers は実走行の incident（operator が「N items × M callees × K iters = 多数 dispatches に膨らむかも」と推定して abort した case）から retroactive に検出され、reviewer feedback iter 1 で Critical-severity finding として上がりやすい。Risks 節 / Reject 列挙だけで mitigation を表現する形は不十分 — **「禁止」「forbidden」** で wording を統一する（"推奨" wording は iter 2 で Major flag になる頻出 finding と同 pattern）。scale-management gate enumeration は **closed list** にして「if a future change introduces a new gate, append it here so the canonical enumeration stays exhaustive」を SKILL.md prose レベルで明示し、新 gate 追加時の追記義務を埋め込む
- **Per-invocation input binding for multi-callee orchestration with structured per-element args**: orchestrator が **per-element loop** で複数 callee を dispatch し、各 callee に **element-specific structured args**（例: `dev-workflow-triage` の per-Finding `Skill(verify-diff)` 呼び出しで Finding 固有の `Description` / `Suggested fix direction`）を渡す設計で、per-dispatch で再度 "verbatim from source body" **per-element record schema** に args 用の field を追加（例: `record.description` / `record.suggested_fix_direction`）、各 field に `default —` + `Bound at (a) entry to verbatim text of <source field>` を明記、(ii) section」と明示、(iii) **callee Inputs ブロックを record-sourced に書き換え**（"verbatim from source" 表記を `record.<field>` 参照に置換、binding step への stable section heading reference 付き — 例: `§ Apply accepted Findings's invocation contract で args を受け取らない場合は binding を **skip 明記**（例: "`publicity-review` does not take these inputs (per its `## Invocation contract`), so this binding applies only to verify-diff" のような 1 - **Per-commit accept gate: render commit body verbatim in a fenced code block, not as a prose promise**: dev-workflow Step 10 の per-commit accept gate で user に commit を提示する際、prose で「Body 含め、diff full preview」のように **「body も含めて出す」と謳いながら、実際の rendering は Subject + files list で止まる** failure mode が観測された場合（user が "bodyはどれですか？" と返す典型 incident）、SKILL.md Step 10 procedure の Present step で **commit body を必ず fenced code block で独立 render** する規律を明記する。canonical pattern: (i) **Subject 用 fenced code block** + (ii) **Body 用 fenced code block**（empty body の場合は `(no body)` プレースホルダ明示）+ (iii) **files list**（pathspec で staging 範囲を明示）+ (iv) **per-file diff**（tracked は `git diff <base-commit>` portion、untracked は `Read` で取得した new-file hunk — 既存ルール `Untracked-file Read fallback for "all-changes diff" presentation` と整合）の 4 要素を **closed list** として SKILL.md prose に列挙する。Body を prose で「含む」と宣言するだけ（実 rendering なし）は **禁止条項** として明示する — Procedure step の wording に「The body MUST appear in a dedicated fenced code block; a prose statement like "body included" without a rendered block is insufficient and triggers an immediate re-render request from the user」のような禁止行為を明文化する。`git commit` の subject + body 構造は git CLI 標準だが、`-m <subject>` 単独で body 無しの commit を作る場合と、`-m <subject> -m <body>` または HEREDOC で body 付き commit を作る場合では accept-gate 上での見え方が異なるため、proposal 段階で body の有無 / 内容を user が approval 判断する materialy として必須レベルに昇格する。1 commit に集約する path（既存ルール `Stop-hook auto-stages unrelated files` の単一 commit specialization）でも 2-commit split path でも、各 commit について Subject + Body + files + diff の 4 要素 closed list は同じ — split の場合は commit ごとに 4 要素セットを repeat する
- **Threshold magic numbers anchored on observable platform signals + buffer ratio (not arbitrary)**: 配布スキルの新 config field で threshold value を default 化する場合、**任意の magic number** を直接書かず、**観測可能な platform signal**（Claude Code の per-file warning 閾値 / API rate limit / built-in size limit 等）に anchor する。canonical pattern: (i) **platform signal を実機で empirically 観測** する（例: `⚠ Large .claude/rules/project.rules.local.md will impact performance (47.6k chars > 40.0k)` から閾値 40000 chars を読む）、(ii) signal をハードコードしないが、**80% 程度の buffer ratio** で trigger threshold を出す（warning が出る前に予防、次回 rules 追加で warning に届かない buffer 確保）、(iii) Configuration table の description に **anchor の説明**（「observed in Claude Code 2.1.x」「80% of upstream X 閾値」等）と **rationale**（「leave headroom for subsequent rule additions」）を明記して将来の閾値変更時に判断材料を残す、(iv) platform signal の version-sensitivity を documenting する（リリースで変わる可能性があるため）。例: `compaction_threshold: 32000` (80% of Claude Code's 40k per-file warning, observed in 2.1.x)。peer review で「磁性的 magic number」と flag される頻出 finding を未然に防ぐ
- **Opt-out via large sentinel value, not boolean disable flag, for numeric threshold config**: 既存の numeric threshold config field（例: `compaction_threshold: 32000`）に **opt-out 経路** を提供する必要が出た場合、**新たな boolean disable flag**（例: `compaction_enabled: false`）を追加するのではなく、**`<field>: 99999999` のような大 sentinel value** を canonical opt-out として宣言する。理由: (i) 新 boolean を追加すると config schema が膨らみ、boolean × numeric の組み合わせ semantics（例: `disable=false but threshold=1000` の挙動）が ambiguous になる、(ii) 既存の numeric pipeline（`if <metric> > threshold then trigger`）に変更を加えず動作するため downstream code が単純に保てる、(iii) 99999999 のような明示的 large value は「実質的に無限大」として読みやすい。Configuration table の description に opt-out path を 1 行明記（例: `Set to a very large number (e.g. \`99999999\`) to opt out`）。CHANGELOG の default-flip 注記でも opt-out 手順は `<field>: <sentinel>` の form で書く。numeric threshold が「閾値超過時に X を起動する」semantic を持つ場合の opt-out には **常に sentinel value 方式**を選び、新 boolean は追加しない
- **Char count vs byte count distinction for multibyte content (`wc -m` vs `wc -c`)**: 配布スキル内 SKILL.md / references / rules files は predominantly Japanese 多言語 text のため、byte 数 (`wc -c`) と char 数 (`wc -m`) が大幅に乖離する（日本語は UTF-8 で 3 bytes/char）。**Claude Code の "Large ... will impact performance" warning は char count 単位** で動作する（`(47.6k chars > 40.0k)` の表示が証拠）。peer reviewer / 外部測定が `wc -c` (byte) で測定して「数値が plan と乖離する」と finding を出した場合、(i) plan の数値が warning message の char 単位からの直接引用であれば正しい、(ii) reviewer の byte 単位測定は誤った unit、(iii) reject 理由として「byte ≠ char (multibyte content); warning message から char count を直接引用済み」と明示。canonical 実装: char count を取る場合は `wc -m <file>` を使う（`wc -c` は禁止）、SKILL.md prose に閾値を埋める際も「chars」単位を明示（「bytes」と書かない）、threshold variable 名も `*_chars` / `*_char_count` 等で suffix する（`*_size` は曖昧で避ける）

- **Experimental feature → opt-in default `false` heuristic (overrides sibling-config consistency)**: 新 config flag を追加する際、既存 sibling config（`interactive_commits` / `task_decomposition` 等）が default `true` (opt-out) でも、追加対象の機能が **直近リリースで導入された experimental 機能** ならデフォルトは `false` (opt-in) に倒す。判断軸: (i) gate される機能が **直近 minor / patch リリース（例: 1 つ前の version）で導入された** experimental 機能か、(ii) 未だ実証実験段階で「特定プロジェクトでのみ走らせて挙動を観察したい」運用フェーズか、(iii) unconditional 走行が想定外の副作用（rule 圧縮 / 自動 fix 等）を起こすリスクが残っているか — いずれかが該当するなら sibling-config consistency より experimental 安全側を優先。Step 4 user-gate で user が opt-in を要求するパターンが頻出するため、Plan 初期作成時に Decisions § N で「default value」を必ず独立 question として立て、Recommendation に opt-in (default `false`) + Alternative に sibling-consistency (default `true`) を並べる。CHANGELOG entry は default-flip rule（`§ Behavior-change default-flip`）を opt-in 方向にも適用 — `**Default: disabled** — set \`<flag>: true\` in \`<config-path>\` to opt in per project` の form で first-line に表記し、prior version で unconditional だった場合は「**Behavior change from v<prior>**: ...users who adopted v<prior>'s <feature> and want to retain that behavior must explicitly set \`<flag>: true\`」を同 bullet 末尾に colocate する
- **Decision insertion (not swap) for previously-unstated default-value choice at Step 4 gate**: 既存ルール `Plan rewrite triggered by user material change at Step 4 gate` の specialization: user が Step 4 gate で要求した material change が「現プランで Decisions に立てていなかった選択肢」（典型例: default value の選択を Recommendation 1 つだけで提示していたが、user が異なる default を要求）の場合、Recommendation/Alternative の **swap** ではなく **新 Decision を Decisions § (N+1) として insert** する。canonical pattern: (a) 新 Decision の見出しに `**user 既選択: <value> — Step 4 gate にて**` annotation を入れ、後追い triage が「user が明示的に選んだ option」と読み取れるようにする、(b) Recommendation に user 選択の value、Alternative に元プランの暗黙 default を並記（既存 swap rule と同じく両 option を visible に保つ）、(c) Approach / Scope / Test plan / Risks も新 default の文脈に sweep 更新（旧 default 由来の Risks は完全置換、新 default 由来の Risks に書き換え）、(d) CHANGELOG entry を新 default 文脈で書き直し（opt-in / opt-out 方向に応じて `**Default: enabled/disabled**` first-line を反転）、(e) 旧プランで存在した「self-application 注意」等の Risk が新 default で neutralize される場合は削除する。swap rule が「既存 Decision の R/A 入れ替え」を扱うのに対し、insertion rule は「Decisions に存在しなかった choice の遡及追加」を扱う — 両者を混同すると、既存 swap rule の wording 通りに無理矢理 swap してしまい元 Decisions の structure を壊す
